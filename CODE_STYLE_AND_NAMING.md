# Code Style And Naming

Эта лекция нужна потому, что качество кода определяется не только корректностью. Очень многое в реальной разработке упирается в читаемость:

- можно ли быстро понять, что делает код
- можно ли быстро найти нужный файл
- не путаются ли имена типов, переменных и функций
- выглядит ли проект как единая система, а не как случайный набор файлов

Ниже практические правила для C++ и Rust на уровне junior.

## Главный принцип именования

Имя должно отвечать на вопрос:

- что это такое
- или что это делает

Плохое имя заставляет читать реализацию.
Хорошее имя уменьшает необходимость лезть внутрь.

## Что плохо почти всегда

- `a`, `b`, `c`, если это не счетчики в очень локальном коде
- `tmp`, `data`, `obj`, `item`, если из контекста неясно, что именно это за сущность
- `processData`, `handleStuff`, `manager`, `helper`, если название ничего не объясняет
- одинаковые имена типа и переменной

Пример плохого кода:

```cpp
Calc Calc(exp);
```

Здесь чтение ломается, потому что имя объекта совпадает с именем типа.

Лучше:

```cpp
Calc calculation(exp);
```

## Что такое хорошее имя переменной

Хорошее имя переменной обычно описывает:

- содержимое
- роль
- иногда единицу измерения или контекст

Плохо:

```cpp
int x;
```

Лучше:

```cpp
int user_count;
```

Или:

```cpp
int max_connections;
```

## Что такое хорошее имя функции

Функция должна называться как действие.

Плохо:

```cpp
void data();
```

Лучше:

```cpp
void print_report();
int count_even_numbers();
bool parse_input();
```

Хорошее имя функции позволяет понять намерение без чтения тела.

## Что такое хорошее имя класса или структуры

Тип данных обычно называют существительным:

- `Task`
- `Book`
- `UserConfig`
- `FileLogger`

Плохо:

- `Data`
- `Manager`
- `Helper`

если из названия неясно, чем сущность реально занимается.

## Когда допустимы короткие имена

Короткие имена допустимы:

- в очень локальном контексте
- для счетчиков цикла
- для математических выражений

Пример:

```cpp
for (std::size_t i = 0; i < values.size(); ++i) {
```

Здесь `i` нормально, потому что область жизни маленькая и роль очевидна.

Но если переменная живет дольше и несет смысл, имя должно быть нормальным.

## Именование в C++

### Практическое правило для junior

Главное не выбрать “идеальный стиль из интернета”, а выбрать один стиль и держать его последовательно.

Нормальный рабочий вариант:

- классы и структуры: `PascalCase`
- функции и переменные: `snake_case`
- приватные поля: `snake_case_`
- константы: `snake_case` или `kPascalCase`, но последовательно

Пример:

```cpp
class TaskManager {
public:
    void add_task(const std::string& title);
    int completed_count() const;

private:
    std::vector<std::string> tasks_;
};
```

### Почему приватные поля часто пишут с `_`

Чтобы быстро отличать:

- локальные переменные
- параметры
- поля объекта

Пример:

```cpp
class User {
public:
    User(std::string name) : name_(std::move(name)) {}

private:
    std::string name_;
};
```

Это снижает двусмысленность.

## Именование файлов в C++

Вот здесь как раз важно не путаться.

### Базовое правило

Имя файла должно соответствовать основной сущности внутри.

Если у тебя есть класс:

- `Task`

то нормальные имена файлов:

- `task.h`
- `task.cpp`

Если есть:

- `TaskManager`

то:

- `task_manager.h`
- `task_manager.cpp`

### Почему так лучше

- файл легко найти по имени сущности
- проект читается предсказуемо
- меньше хаоса в include и навигации

### Какой стиль имени файла лучше

Для C++ на твоем уровне я рекомендую:

- `snake_case` для файлов

То есть:

- `task_manager.h`
- `student_record.h`
- `file_logger.cpp`

А не:

- `TaskManager.cpp`
- `StudentRecord.hpp`

Не потому, что второй вариант запрещен, а потому что `snake_case` проще и спокойнее выглядит в проекте, особенно на Linux.

## `.h` или `.hpp`

Оба варианта используются.

### Практический совет

Для учебных и небольших прикладных проектов выбери один стиль и не мешай.

Если стартуешь с нуля, вполне нормально:

- использовать `.h` для заголовков
- `.cpp` для реализаций

То есть:

- `task.h`
- `task.cpp`

Этого достаточно.

### Когда бывает `.hpp`

`.hpp` часто используют, чтобы подчеркнуть “это именно C++ заголовок, а не C”.

Но для твоего уровня это не ключевой вопрос. Ключевой вопрос - последовательность внутри проекта.

## Как раскладывать C++ проект по файлам

Для небольшого проекта уже достаточно такой структуры:

- `include/`
- `src/`
- `tests/`

Пример:

```text
project/
  CMakeLists.txt
  README.md
  include/
    task.h
    task_manager.h
  src/
    main.cpp
    task.cpp
    task_manager.cpp
  tests/
    task_manager_tests.cpp
```

### Что класть в `.h`

Обычно:

- объявления классов
- объявления функций
- интерфейс

### Что класть в `.cpp`

Обычно:

- реализации функций
- детали логики

## Что не надо делать в C++ проекте

- держать весь код в `main.cpp`
- называть файлы `a.cpp`, `test2.cpp`, `newfile.cpp`
- использовать разные стили имен одновременно без причины
- называть класс `Vector`, если рядом уже есть `std::vector` и возникает путаница по смыслу

## Именование в Rust

У Rust есть довольно устоявшиеся идиомы.

### Что обычно принято

- типы: `PascalCase`
- функции: `snake_case`
- переменные: `snake_case`
- модули и файлы: `snake_case`
- константы: `SCREAMING_SNAKE_CASE`

Пример:

```rust
struct TaskManager {
    tasks: Vec<String>,
}

fn count_even_numbers(values: &[i32]) -> usize {
    0
}

const DEFAULT_PORT: u16 = 8080;
```

## Почему в Rust файлы почти всегда `snake_case`

Потому что модульная система языка и общее сообщество к этому сильно привыкли.

Пример:

- `task.rs`
- `task_manager.rs`
- `storage.rs`
- `error.rs`

Это наиболее естественный стиль для Rust-проектов.

## Как раскладывать Rust проект

Для небольшого проекта:

```text
project/
  Cargo.toml
  src/
    main.rs
    lib.rs
    task.rs
    task_manager.rs
    storage.rs
    error.rs
  tests/
    cli_tests.rs
```

### `main.rs`

Точка входа бинарного приложения.

### `lib.rs`

Общий модуль библиотеки проекта, если ты хочешь разложить логику как библиотечную часть.

### Остальные файлы

Обычно называются по основной сущности или роли:

- `parser.rs`
- `storage.rs`
- `config.rs`
- `error.rs`

## Как выбирать имя файла

Полезный вопрос:

- какая главная сущность или ответственность у этого файла

Если файл хранит код про хранение данных, имя `storage.rs` лучше, чем `utils.rs`.

Если файл содержит доменную модель задачи, имя `task.rs` или `book.rs` лучше, чем `data.rs`.

## Что такое плохое имя файла

Плохие имена:

- `utils.cpp`
- `helpers.rs`
- `data.cpp`
- `misc.rs`

Почему плохие:

- не объясняют границу ответственности
- быстро превращаются в свалку несвязанных функций

Иногда такие файлы неизбежны временно, но как постоянная структура это плохой знак.

## Именование тестов

В C++:

- `task_manager_tests.cpp`
- `parser_tests.cpp`

В Rust:

- `parser_tests.rs`
- `cli_tests.rs`

Имя тестового файла тоже должно показывать, что именно тестируется.

## Практические правила на каждый день

1. Имя должно объяснять смысл, а не прятать его.
2. Типы называй как сущности, функции как действия.
3. Имя файла должно соответствовать основной сущности или ответственности.
4. Держи один стиль по всему проекту.
5. Если имя звучит слишком общо, скорее всего оно плохое.

## Что лучше для тебя прямо сейчас

### Для C++

- классы: `PascalCase`
- функции и переменные: `snake_case`
- приватные поля: `name_`
- файлы: `snake_case.h` и `snake_case.cpp`

### Для Rust

- типы: `PascalCase`
- функции и переменные: `snake_case`
- константы: `SCREAMING_SNAKE_CASE`
- файлы и модули: `snake_case.rs`
