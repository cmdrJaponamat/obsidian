# Лекция 2. Память, ссылки и время жизни объектов

## Почему эта тема ключевая

Одна из главных причин, почему C++ считается сложным, это не синтаксис, а управление временем жизни объектов. На junior уровне от тебя не ждут редкой магии, но ждут, что ты не будешь случайно ломать память.

## Stack и heap

### Stack

Локальные переменные функции обычно живут на стеке.

```cpp
void run() {
    int x = 10;
}
```

Когда функция завершилась, `x` уничтожается автоматически.

### Heap

Память можно выделять динамически. Исторически это делали через `new`.

```cpp
int* value = new int(42);
delete value;
```

Но в современном C++ прямое `new/delete` почти всегда лучше избегать в прикладном коде. Причина проста: слишком легко забыть освобождение, удалить дважды или перепутать владельца ресурса.

## Время жизни объекта

У каждого объекта есть:

- момент создания
- период существования
- момент уничтожения

Пример:

```cpp
{
    std::string name = "Alex";
} // здесь name уничтожается
```

Нужно мысленно всегда отслеживать:

- кто владеет объектом
- кто только смотрит на него
- не используется ли объект после уничтожения

## Ссылки

Ссылка - это альтернативное имя существующего объекта.

```cpp
int value = 5;
int& ref = value;
ref = 10;
```

Теперь `value == 10`.

### Когда ссылки полезны

- передать объект в функцию без копии
- изменить исходный объект внутри функции
- сделать интерфейс понятнее, чем через указатель, если `null` не допускается

## Указатели

Указатель хранит адрес объекта.

```cpp
int value = 7;
int* ptr = &value;
std::cout << *ptr << '\n';
```

Указатели нужны, но на junior уровне важно не злоупотреблять ими там, где достаточно:

- обычных объектов
- ссылок
- контейнеров STL
- умных указателей

## Основные ошибки с памятью

### Висячая ссылка или указатель

```cpp
const std::string& bad_ref() {
    std::string text = "hello";
    return text;
}
```

Это ошибка: функция возвращает ссылку на уже уничтоженный объект.

### Утечка памяти

```cpp
void leak() {
    int* data = new int(5);
}
```

`delete` не вызван, память потеряна.

### Двойное освобождение

```cpp
int* p = new int(3);
delete p;
delete p;
```

Так делать нельзя.

## RAII

RAII - один из центральных принципов C++.

Смысл:

- ресурс захватывается в конструкторе объекта
- ресурс освобождается в деструкторе

Ресурсом может быть:

- память
- файл
- socket
- mutex

Пример с файлом на уровне идеи:

```cpp
class FileLogger {
public:
    FileLogger(const std::string& path);
    ~FileLogger();

    void log(const std::string& message);
};
```

Когда объект `FileLogger` выходит из области видимости, файл закрывается автоматически.

Это полезно потому, что освобождение не зависит от того, забыли ли ты вызвать отдельную функцию вручную.

## Конструктор и деструктор

```cpp
class Counter {
public:
    Counter() {
        std::cout << "Created\n";
    }

    ~Counter() {
        std::cout << "Destroyed\n";
    }
};
```

Конструктор создает корректное состояние объекта. Деструктор освобождает ресурсы.

## Копирование и перемещение

Если объект владеет ресурсом, копирование становится чувствительной темой.

Представь класс с сырым указателем:

```cpp
class Buffer {
public:
    Buffer(size_t size) : size_(size), data_(new int[size]) {}
    ~Buffer() { delete[] data_; }

private:
    size_t size_;
    int* data_;
};
```

Если такой объект скопировать по умолчанию, два объекта будут думать, что владеют одним и тем же массивом. Итог - неопределенное поведение.

Отсюда появляются:

- rule of three
- rule of five

На junior уровне важно не столько заучить формулировку, сколько понять смысл:

- если у класса есть владение ресурсом, нужно явно подумать о копировании, перемещении и уничтожении

## `std::unique_ptr`

Это основной умный указатель, который выражает эксклюзивное владение.

```cpp
auto ptr = std::make_unique<int>(42);
```

Смысл:

- ресурсом владеет один объект
- при уничтожении `unique_ptr` ресурс освобождается автоматически
- копировать нельзя, перемещать можно

Это очень хороший default choice, когда объект должен владеть чем-то динамически созданным.

## `std::shared_ptr`

Используется, когда владение действительно совместное.

```cpp
auto shared = std::make_shared<std::string>("data");
```

Но проблема в том, что многие новички используют `shared_ptr` как "универсальный костыль". Это плохая практика.

Используй его только если правда есть несколько владельцев с общей ответственностью за lifetime.

## Move semantics

Перемещение позволяет передать ресурсы от одного объекта другому без дорогостоящего копирования.

Простой смысл:

- копирование создает независимую копию ресурса
- перемещение передает владение

Пример:

```cpp
std::vector<int> create_data() {
    std::vector<int> data = {1, 2, 3, 4};
    return data;
}
```

Современный C++ умеет делать это эффективно за счет move semantics и оптимизаций возврата.

## Практические правила junior-разработчика

- сначала используй обычные объекты и STL-контейнеры
- если нужна динамика владения, смотри в сторону `unique_ptr`
- не трогай `new/delete`, если задачу можно решить стандартными средствами
- думай, кто владеет ресурсом
- не возвращай ссылки на локальные объекты
