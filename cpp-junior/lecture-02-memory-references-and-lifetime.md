# Лекция 2. Память, стек, куча, ссылки и время жизни объектов

## Почему эту лекцию надо понять по-настоящему

Если в первой лекции ты мог выжить на синтаксисе, то здесь поверхностное понимание уже не сработает. Большая часть "страшности" C++ связана с тем, что программист должен понимать, где живут данные, кто ими владеет и когда они уничтожаются.

## Что вообще такое память процесса

Когда программа запускается, операционная система выделяет ей адресное пространство. Упрощенно можно думать о нем как о большом наборе адресов, доступных процессу.

В этой памяти размещаются:

- код программы
- глобальные данные
- стек
- куча
- другие области, нужные рантайму и библиотекам

Важно: стек и куча - это не "разные микросхемы" и не "разные физические места на компьютере". Это разные логические области памяти процесса, которые используются по разным правилам.

## Стек

Стек обычно используется для:

- локальных переменных функции
- параметров вызова
- служебной информации о вызове функций

Пример:

```cpp
void run() {
    int x = 10;
    double y = 5.5;
}
```

`x` и `y` обычно размещаются на стеке.

### Почему стек удобен

- выделение и освобождение очень быстрые
- lifetime локальных объектов связан с блоком или вызовом функции
- программисту не нужно освобождать такую память вручную

### Как работает lifetime на стеке

Когда выполнение входит в функцию, создается кадр стека. Когда функция завершается, этот кадр уничтожается.

Из-за этого локальные объекты живут только пока активен их блок.

```cpp
int main() {
    int x = 5;
} // здесь x уничтожается
```

### Ограничения стека

Стек не бесконечен. Его размер обычно ограничен настройками ОС и процесса. Точное значение зависит от платформы, запускающей среды и конфигурации.

Практический вывод:

- на стеке удобно держать обычные локальные объекты разумного размера
- огромные массивы на стеке создавать опасно

Плохая идея:

```cpp
int huge[100000000];
```

Такой код может привести к переполнению стека.

## Куча

Куча используется для динамической памяти, когда размер или lifetime данных нельзя удобно уложить в обычную локальную переменную.

Исторический пример:

```cpp
int* value = new int(42);
delete value;
```

### В чем смысл кучи

Куча полезна, когда:

- объект должен жить дольше текущей функции
- размер данных известен только во время выполнения
- нужна сложная структура данных с динамическим lifetime

### Цена кучи

Динамическое выделение памяти:

- обычно дороже, чем работа со стеком
- требует очень четкого понимания владения
- легко приводит к утечкам и повреждению памяти при ошибках

## Почему `new/delete` сегодня стараются избегать

Важно не перепутать: не потому, что они "запрещены", а потому, что ручное управление памятью слишком легко сделать неправильным.

Основные риски:

- забыть `delete`
- удалить один и тот же объект дважды
- продолжить использовать указатель после `delete`
- не понимать, кто именно отвечает за освобождение памяти

## Что делать вместо прямого `new/delete`

Вот правильный современный порядок предпочтений:

### 1. Обычные локальные объекты

Если можно создать объект просто как переменную, это часто лучший вариант.

```cpp
std::string name = "Ivan";
std::vector<int> numbers = {1, 2, 3};
```

### 2. Контейнеры STL

Если нужен динамический размер, часто достаточно контейнера, который сам управляет памятью.

```cpp
std::vector<int> data(1000);
```

Ты получаешь динамический массив, но не управляешь сырой памятью вручную.

### 3. Умные указатели

Если реально нужен объект с динамическим lifetime, чаще всего смотри на:

- `std::unique_ptr`
- реже `std::shared_ptr`

### 4. Прямой `new/delete`

Обычно только в специальных случаях, когда ты хорошо понимаешь, зачем именно это делаешь.

Для junior это почти никогда не лучший первый выбор.

## Ссылки

Ссылка - это не новый объект, а альтернативное имя уже существующего объекта.

```cpp
int value = 10;
int& ref = value;
ref = 20;
```

Теперь `value == 20`.

### Что важно о ссылках

- ссылка должна быть привязана к объекту при создании
- обычную ссылку нельзя "переназначить" на другой объект
- ссылка не может быть `null`

Именно поэтому ссылка часто удобнее указателя там, где отсутствие объекта не допускается.

## Указатели

Указатель хранит адрес объекта.

```cpp
int value = 7;
int* ptr = &value;
std::cout << *ptr << '\n';
```

### Чем указатель отличается от ссылки

Указатель:

- может быть `nullptr`
- может менять адрес, который хранит
- явно требует разыменования через `*`

Ссылка:

- должна ссылаться на реальный объект
- синтаксически выглядит как обычный объект
- удобнее для выражения "объект точно есть"

## Время жизни объекта

Время жизни, или lifetime, - это интервал между созданием и уничтожением объекта.

Это одна из главных тем C++.

Пример:

```cpp
{
    std::string text = "hello";
}
```

После выхода из блока объект `text` уничтожается. Любая ссылка или указатель на него после этого становятся недействительными.

## Висячие ссылки и указатели

Пример ошибки:

```cpp
const std::string& bad() {
    std::string text = "hello";
    return text;
}
```

Почему это плохо:

- `text` создается внутри функции
- при выходе из функции `text` уничтожается
- возвращаемая ссылка указывает на уже несуществующий объект

Это и есть висячая ссылка.

## RAII

RAII означает: ресурс привязан к объекту, а освобождение ресурса происходит автоматически при уничтожении объекта.

Это основа хорошего C++.

### Что может быть ресурсом

- память
- файл
- сокет
- блокировка mutex

### Главное достоинство RAII

Освобождение ресурса происходит автоматически, даже если:

- функция завершилась раньше из-за `return`
- возникло исключение
- код разросся и стал сложнее

То есть ресурс не "забывается" вручную.

## Конструктор и деструктор

Конструктор подготавливает объект к работе. Деструктор освобождает ресурсы объекта.

```cpp
class FileLogger {
public:
    FileLogger();
    ~FileLogger();
};
```

Если класс владеет ресурсом, деструктор обычно играет очень важную роль.

## Владение ресурсом

Ключевой вопрос почти в любой теме о памяти:

Кто владеет объектом?

То есть:

- кто отвечает за его lifetime
- кто должен освобождать ресурс
- кто просто временно пользуется объектом

Если на этот вопрос нет ясного ответа, код почти наверняка станет проблемным.

## `std::unique_ptr`

Это умный указатель для эксклюзивного владения.

```cpp
auto ptr = std::make_unique<int>(42);
```

Что это дает:

- память освобождается автоматически
- владелец ровно один
- ownership в коде выражен явно

Это один из лучших инструментов современного C++ для объектов с динамическим lifetime.

## `std::shared_ptr`

Это умный указатель для совместного владения.

```cpp
auto ptr = std::make_shared<std::string>("hello");
```

Но с ним нужно быть осторожным.

### Почему его не стоит брать "по умолчанию"

Потому что совместное владение:

- сложнее по смыслу
- может прятать архитектурную неясность
- иногда приводит к циклическим ссылкам и неожиданному lifetime

Если можно использовать обычный объект или `unique_ptr`, это часто лучше.

## Копирование и ресурс

Если объект владеет ресурсом, копирование нельзя оставлять бездумно.

Проблемный пример:

```cpp
class Buffer {
public:
    Buffer(std::size_t size) : size_(size), data_(new int[size]) {}
    ~Buffer() { delete[] data_; }

private:
    std::size_t size_;
    int* data_;
};
```

Если такой объект скопировать "как есть", будет скопирован указатель, а не сам массив. Тогда два объекта начнут думать, что владеют одной и той же памятью.

Отсюда проблемы:

- двойное освобождение
- повреждение памяти

Именно поэтому владение ресурсом заставляет думать о специальных функциях класса и о rule of three/five.

## Перемещение

Во многих случаях ресурс не нужно копировать, его можно передать.

Это и есть идея move semantics.

Интуитивно:

- копия создает еще один независимый ресурс
- перемещение передает уже существующий ресурс новому владельцу

Позже это будет особенно важно для `std::string`, `std::vector` и пользовательских классов.

## Практические правила для старта

- если можно, создавай обычный объект, а не работай через указатель
- если нужен динамический размер, сначала смотри на `std::vector` и другие контейнеры
- если нужен динамический lifetime, чаще всего смотри на `std::unique_ptr`
- не используй прямой `new/delete`, если можно решить задачу стандартными средствами
- всегда задавай вопрос: кто владеет объектом

## Что нужно вынести из лекции

- стек и куча - это разные логические области памяти процесса, а не два "типа железа"
- стек удобен для локальных объектов и ограничен по размеру
- куча нужна для динамического lifetime, но требует большей дисциплины
- ручной `new/delete` опасен не сам по себе, а тем, что слишком легко ошибиться
- modern C++ предпочитает обычные объекты, STL-контейнеры, RAII и умные указатели
