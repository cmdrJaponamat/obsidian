# Лекция 3. STL, классы и как организовывать код без каши

## Что здесь главное

После первых программ новичок часто попадает в ловушку:

- либо все пишет вручную слишком низкоуровнево
- либо начинает без понимания плодить классы и "ООП ради ООП"

Нужен более спокойный и практичный взгляд:

- использовать стандартную библиотеку там, где она уже решает задачу
- делать классы там, где реально есть сущность и инварианты
- раскладывать проект по файлам по смыслу, а не случайно

## Что такое STL на практическом уровне

STL - это не один контейнер и не "магия". Это набор стандартных инструментов:

- контейнеров
- алгоритмов
- итераторов
- полезных типов

На junior-уровне не нужно знать все. Нужно хорошо знать небольшой рабочий минимум.

## `std::vector`

Это основной контейнер для последовательностей.

Когда он подходит:

- есть список элементов
- важен доступ по индексу
- нужно часто идти по всем элементам

```cpp
std::vector<std::string> tasks;
tasks.push_back("read");
tasks.push_back("code");
```

### Почему `vector` так важен

- динамически растет
- сам управляет памятью
- хорошо поддерживается стандартными алгоритмами
- подходит для огромного числа прикладных задач

Очень часто новичок зря пытается "сделать свой массив", хотя `vector` уже решает задачу лучше и безопаснее.

## `std::string`

Хотя это не контейнер в учебном смысле, на практике это один из центральных типов.

Он:

- хранит последовательность символов
- управляет своей памятью
- умеет копироваться и перемещаться
- дает удобный интерфейс

Это одна из причин, почему в современном C++ часто можно вообще не работать с сырыми C-строками на junior-уровне.

## `std::map` и `std::unordered_map`

Обе структуры хранят пары ключ-значение, но с разными свойствами.

### `std::map`

- обычно хранит ключи в отсортированном порядке
- операции часто имеют логарифмическую сложность

### `std::unordered_map`

- основан на хешировании
- часто дает очень быстрый доступ по ключу
- порядок элементов не гарантирован

Для большинства прикладных junior-задач на "подсчитать частоты" и "хранить индекс по id" `unordered_map` очень удобен.

## `std::optional`

Иногда значение может быть, а может не быть, и это не ошибка.

Пример:

- поиск пользователя по id
- поиск элемента в коллекции

Вместо того чтобы возвращать "магическое значение" вроде `-1`, лучше явно выразить отсутствие результата.

```cpp
std::optional<int> find_index(...);
```

Это делает интерфейс честнее.

## Алгоритмы STL

Очень полезно как можно раньше привыкнуть не только к контейнерам, но и к стандартным алгоритмам.

Минимальный набор:

- `std::sort`
- `std::find_if`
- `std::count_if`
- `std::transform`

Почему это лучше, чем вручную каждый раз писать цикл:

- код короче
- намерение яснее
- меньше типового шума
- меньше шансов ошибиться в рутине

## Что такое класс

Класс - это способ объединить:

- данные
- операции над этими данными
- правила корректного использования

Пример:

```cpp
class BankAccount {
public:
    explicit BankAccount(int balance) : balance_(balance) {}

    void deposit(int amount) {
        balance_ += amount;
    }

    int balance() const {
        return balance_;
    }

private:
    int balance_;
};
```

Здесь класс не просто "упаковал переменную". Он задал сущность с понятным интерфейсом.

## Инкапсуляция

Инкапсуляция означает, что внутреннее состояние объекта не раздается наружу без контроля.

Почему это полезно:

- объект защищает свои инварианты
- внешний код не ломает внутреннее состояние случайно
- проще менять реализацию, не меняя интерфейс

Если все поля публичные и меняются как попало, класс часто теряет смысл.

## Что такое инвариант

Инвариант - это условие, которое должно оставаться истинным для корректного объекта.

Например, для банковского счета можно хотеть, чтобы баланс не становился отрицательным без специальных правил.

Хороший класс помогает поддерживать такие свойства.

## Композиция и наследование

Новички часто слышат "ООП" и сразу думают про наследование. Это ошибка.

### Композиция

Композиция означает, что один объект содержит другие.

Пример:

- `Library` содержит `std::vector<Book>`
- `TaskManager` содержит `Storage`

Это очень частый и здоровый способ строить программу.

### Наследование

Наследование полезно, когда одна сущность действительно является разновидностью другой и тебе нужен общий интерфейс.

Но если наследование добавлено только ради "красивой архитектуры", это обычно лишняя сложность.

## Интерфейс и реализация

Хороший дизайн часто разделяет:

- что умеет сущность
- как именно она это делает

В C++ это часто выражается через `.h` и `.cpp`.

### Заголовочный файл

Обычно содержит объявления:

- классов
- функций
- интерфейсов

### Исходный файл

Обычно содержит реализации.

Это полезно потому, что:

- код легче читать
- проект проще поддерживать
- интерфейс отделяется от деталей

## Как не превратить проект в хаос

Очень типичный плохой путь:

- один огромный `main.cpp`
- чтение ввода, логика, работа с файлами и вывод смешаны
- функции по 80 строк
- имена вроде `processData2`

Более здоровый путь:

- сущности предметной области отдельно
- логика отдельно
- ввод/вывод отдельно
- storage отдельно

## Как понять, нужен ли класс

Полезные вопросы:

- есть ли здесь сущность со своим состоянием?
- есть ли у нее понятные операции?
- есть ли инварианты, которые нужно защищать?

Если ответы "нет", возможно, достаточно обычной функции и структуры данных.

## Что нужно вынести из лекции

- STL надо использовать активно, а не обходить стороной
- `vector`, `string`, `unordered_map` и базовые алгоритмы закрывают очень много задач
- класс нужен не для галочки, а когда есть состояние, поведение и инварианты
- композиция чаще полезнее наследования
- разделение по файлам и слоям делает код понятнее
