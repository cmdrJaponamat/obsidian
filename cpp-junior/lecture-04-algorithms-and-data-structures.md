# Лекция 4. Алгоритмы и структуры данных для junior C++

## Зачем это нужно

Даже если вакансия прикладная, на собеседовании часто проверяют, умеешь ли ты:

- выбрать нормальную структуру данных
- оценить сложность
- решить небольшую задачу без полного хаоса

Не нужен уровень олимпиадника. Нужен уровень человека, который понимает базовые паттерны.

## Асимптотическая сложность

Это способ оценить, как растет стоимость алгоритма при увеличении входных данных.

Чаще всего нужны:

- `O(1)`
- `O(log n)`
- `O(n)`
- `O(n log n)`
- `O(n^2)`

### Примеры

- доступ к элементу `vector` по индексу: обычно `O(1)`
- линейный поиск по массиву: `O(n)`
- сортировка: обычно `O(n log n)`
- двойной вложенный цикл по массиву: часто `O(n^2)`

## Основные структуры данных

### `std::vector`

Используй, когда:

- нужен упорядоченный список
- часто идешь по всем элементам
- нужен доступ по индексу

### `std::unordered_map`

Используй, когда:

- нужен поиск по ключу
- считаешь частоты
- хочешь быстро понимать, встречалось ли значение

### `std::set` / `std::unordered_set`

Используй, когда:

- нужно хранить уникальные значения
- нужно быстро проверять наличие

### `stack` и `queue`

Полезны для задач:

- скобки
- обходы
- обработка в порядке FIFO/LIFO

## Типовые паттерны

### Два указателя

Подходит для:

- работы с отсортированными массивами
- задач на подотрезки
- удаление дублей

### Sliding window

Подходит для:

- поиска лучшего окна фиксированной или меняющейся длины
- подсчета по подмассивам

### Prefix sums

Подходит для:

- быстрых сумм по диапазонам

### Binary search

Подходит не только для поиска элемента, но и для "поиска по ответу", если есть монотонность условия.

## Пример: частоты слов

```cpp
std::unordered_map<std::string, int> count_words(
    const std::vector<std::string>& words) {
    std::unordered_map<std::string, int> result;
    for (const auto& word : words) {
        result[word]++;
    }
    return result;
}
```

Почему это хороший junior-код:

- структура данных выбрана осознанно
- решение линейное по числу слов
- нет лишней сложности

## Пример: проверка корректности скобок

Идея:

- открывающую скобку кладем в стек
- закрывающую сверяем с верхушкой

Это типичная задача на структуру данных, а не на синтаксический трюк.

## Как думать на собеседовании

Плохой порядок:

1. начать писать код
2. запутаться
3. потом думать

Лучший порядок:

1. кратко проговорить задачу
2. выбрать структуру данных
3. оценить сложность
4. только потом писать код

## Что обычно портит решение

- выбран не тот контейнер
- лишняя вложенность циклов
- не обработаны пустые случаи
- переменные названы бессмысленно
- код не разбит на читаемые шаги

## Минимальный набор задач, который надо реально прорешать

- palindrome
- two sum
- valid parentheses
- merge sorted arrays
- top frequent words
- max sum of fixed-size window
- basic binary search
