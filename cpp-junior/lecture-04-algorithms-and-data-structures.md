# Лекция 4. Алгоритмическое мышление и выбор структур данных

## Почему алгоритмы нужны даже прикладному junior

Многие думают, что алгоритмы нужны только для олимпиад. На практике работодатель чаще проверяет не редкие трюки, а способность:

- не писать заведомо плохое решение
- выбрать нормальную структуру данных
- объяснить сложность
- увидеть, где задача решается в один проход, а где ты случайно делаешь квадрат

## Что такое сложность алгоритма

Когда говорят `O(n)` или `O(n log n)`, обычно имеют в виду, как растет объем работы алгоритма при росте размера входных данных.

Это не "точное время в миллисекундах", а грубая модель роста.

## Самые важные оценки

### `O(1)`

Работа не растет вместе с размером входа.

Пример:

- доступ к элементу `vector` по индексу

### `O(n)`

Работа растет линейно.

Пример:

- пройти по всем элементам массива

### `O(n^2)`

Работа растет квадратично.

Пример:

- двойной цикл по одному и тому же массиву

### `O(log n)` и `O(n log n)`

Часто встречаются в поиске и сортировке.

Для junior пока важнее узнавать их как хорошие рабочие классы сложности, чем доказывать все формально.

## Почему сложность важна

Решение может быть:

- корректным
- простым
- но слишком медленным

Если на маленьких входах это незаметно, на больших программа может стать непригодной.

## Выбор структуры данных

Один из самых частых признаков хорошего решения: правильно выбрана структура данных.

### `vector`

Подходит, когда:

- данные лежат последовательно
- нужен индекс
- нужен проход по всем элементам

### `unordered_map`

Подходит, когда:

- нужен быстрый доступ по ключу
- нужно считать частоты
- нужно быстро понимать, встречался ли элемент

### `set` или `unordered_set`

Подходят, когда:

- нужно хранить только уникальные значения
- нужно быстро проверять наличие элемента

### `stack`

Подходит, когда:

- нужен принцип "последним пришел, первым ушел"
- задачи на скобки, откат, вложенность

### `queue`

Подходит, когда:

- нужен принцип "первым пришел, первым ушел"
- обработка задач по очереди

## Типовой процесс решения задачи

Очень полезно приучить себя к такому порядку:

1. Сформулировать задачу простыми словами.
2. Понять, что именно нужно хранить.
3. Выбрать структуру данных.
4. Оценить сложность.
5. Только потом писать код.

Новичок часто делает наоборот: начинает печатать цикл, а думает потом. Это сильно повышает шанс запутаться.

## Пример: подсчет частоты слов

Задача:

- есть набор слов
- нужно узнать, сколько раз встречается каждое

Подход:

- нужен ключ = слово
- нужно значение = счетчик

Значит, `unordered_map<std::string, int>` - естественный выбор.

```cpp
std::unordered_map<std::string, int> count_words(
    const std::vector<std::string>& words) {
    std::unordered_map<std::string, int> counts;

    for (const std::string& word : words) {
        counts[word]++;
    }

    return counts;
}
```

## Пример: поиск пары чисел с заданной суммой

Если делать двумя вложенными циклами, получится `O(n^2)`.

Но если хранить уже увиденные числа в хеш-таблице, можно решать задачу быстрее.

Это и есть алгоритмическое мышление в прикладном смысле:

- не магия
- не олимпиадность
- а выбор правильной структуры данных

## Типовые паттерны

### Two pointers

Полезен, когда:

- работаешь с отсортированным массивом
- анализируешь диапазон

### Sliding window

Полезен, когда:

- ищешь лучший подотрезок
- работаешь с окнами фиксированной или меняющейся длины

### Prefix sums

Полезен, когда:

- нужно быстро считать суммы на диапазонах

### Binary search

Полезен не только для "найти элемент", но и для задач с монотонным условием.

## Что значит "правильное" решение на собеседовании

Обычно это не самое короткое решение, а такое, где ты можешь сказать:

- какую структуру данных выбрал
- почему именно ее
- какая будет сложность
- какие крайние случаи учел

Это производит намного лучшее впечатление, чем молча написанный код без объяснения.

## Типичные ошибки

- хранить данные в неподходящей структуре
- писать вложенные циклы там, где можно за один проход
- не думать о пустом вводе
- не проговаривать сложность
- путать корректность и эффективность

## Что нужно вынести из лекции

- алгоритмическая база нужна не ради спорта, а ради адекватных решений
- сначала выбирается структура данных, потом пишется код
- `vector`, `unordered_map`, `set`, `stack`, `queue` закрывают огромную часть junior-задач
- даже базовая оценка сложности уже сильно повышает качество решений
