# Лекция 4. Алгоритмы и структуры данных: как выбирать не по памяти, а по смыслу

## Почему эта лекция должна быть глубже списка контейнеров

Если просто запомнить:

- "unordered_map быстрый"
- "vector хороший"
- "binary search полезен"

то при первой же нестандартной задаче это знание рассыпается. Чтобы реально использовать алгоритмы, нужно видеть:

- какую форму имеют данные
- какие операции над ними частые
- сколько раз ты по ним проходишь
- как выбор структуры данных влияет на сложность

## Что такое сложность и зачем она вообще нужна

Когда говорят `O(n)` или `O(n^2)`, речь не о точном времени в секундах, а о том, как растет объем работы при увеличении входа.

Почему это важно:

- два решения могут быть оба корректны
- но одно будет нормально работать на 10 элементах и ломаться на 100000

## Базовые оценки

### `O(1)`

Стоимость операции не растет вместе с размером входа.

Пример:

- доступ к `vector[i]`

### `O(n)`

Стоимость растет линейно.

Пример:

- пройти по всем элементам один раз

### `O(n^2)`

Стоимость растет квадратично.

Пример:

- для каждого элемента снова пройти по всем элементам

### `O(log n)`

Часто возникает там, где на каждом шаге мы отбрасываем большую часть вариантов.

Пример:

- бинарный поиск

### `O(n log n)`

Часто встречается у хороших сортировок.

## Почему структура данных меняет сложность

Представь задачу:

- есть список слов
- нужно понять, встречалось ли слово раньше

Если хранить все в `vector`, каждый раз поиск будет линейным.

Если хранить в `unordered_set`, проверка наличия будет обычно быстрее.

Значит, одна и та же прикладная задача может менять сложность только из-за выбора контейнера.

## Как думать до кода

Полезный порядок:

1. Что нужно хранить?
2. Что с этими данными нужно делать чаще всего?
3. Нужен ли порядок?
4. Нужен ли быстрый доступ по ключу?
5. Нужны ли уникальные элементы?

Только после этого имеет смысл писать код.

## Выбор структуры данных на примерах

### Случай 1. Просто список чисел

Нужны:

- хранение
- проход по всем элементам
- иногда доступ по индексу

Естественный выбор: `std::vector<int>`

### Случай 2. Частоты слов

Нужно:

- слово
- сколько раз встретилось

Значит:

- ключ = слово
- значение = счетчик

Естественный выбор: `std::unordered_map<std::string, int>`

Почему не `unordered_set`:

- `set` и `unordered_set` хранят только сам факт наличия
- а здесь нужно еще число повторений

Пример:

```cpp
std::unordered_map<std::string, int> count_words(
    const std::vector<std::string>& words) {
    std::unordered_map<std::string, int> counts;

    for (const std::string& word : words) {
        counts[word]++;
    }

    return counts;
}
```

### Случай 3. Удалить дубли

Если тебе нужно просто хранить уникальные элементы, без счетчика:

- `std::set`
- или `std::unordered_set`

Здесь уже `unordered_set` может быть хорошим выбором.

## `map` против `unordered_map`

### `map`

Обычно дерево.

Плюсы:

- порядок ключей сохраняется
- можно итерироваться в отсортированном порядке

Минусы:

- операции обычно `O(log n)`

### `unordered_map`

Обычно хеш-таблица.

Плюсы:

- очень удобен для словарей и частот
- доступ обычно быстрый

Минусы:

- порядок не гарантирован

Значит, вопрос "какой лучше" неверен сам по себе. Правильный вопрос:

- что тебе важнее: порядок или быстрый словарь без порядка

## Почему наивное решение бывает плохим

Возьмем задачу: найти пару чисел с суммой `target`.

### Наивный подход

Для каждого элемента проверить все остальные.

Это:

- просто
- корректно
- но `O(n^2)`

### Улучшение

Хранить уже увиденные числа в хеш-таблице или множестве.

Тогда можно за один проход:

- для текущего числа смотреть, встречался ли нужный комплемент

Именно так структура данных меняет решение.

## Паттерны решения задач

Сейчас важно не заучивать названия, а понимать типовую идею.

## Two pointers

Обычно применяется, когда:

- данные отсортированы
- нужно двигать левую и правую границу

### Типичная задача

Есть отсортированный массив. Нужно найти, есть ли пара с суммой `target`.

Идея:

- один указатель слева
- другой справа
- если сумма слишком мала, двигаем левый
- если слишком велика, двигаем правый

Почему это работает:

- массив отсортирован
- изменение одного из указателей предсказуемо меняет сумму

## Sliding window

Применяется, когда работа идет по "окну" подмассива или подстроки.

### Типичная задача

Найти максимальную сумму подмассива фиксированной длины `k`.

### Наивное решение

Для каждого окна заново суммировать все `k` элементов.

### Улучшение

Если окно сдвинулось на один элемент:

- один элемент ушел
- один пришел

Значит, новую сумму можно получить из старой:

- вычесть уходящий
- добавить входящий

Вот это и есть sliding window на интуитивном уровне.

## Prefix sums

Подходит, когда:

- часто спрашивают суммы по диапазонам

### Идея

Сначала строим массив префиксных сумм:

- `prefix[i]` хранит сумму от начала до позиции `i`

Тогда сумма на диапазоне считается не полным проходом, а разностью двух префиксов.

То есть:

- один предварительный проход
- потом быстрые ответы на много запросов

## Binary search

### Классический случай

Есть отсортированный массив, нужно найти элемент.

Идея:

- берем середину
- понимаем, в какой половине еще может быть ответ
- вторую половину отбрасываем

Почему это быстро:

- на каждом шаге число кандидатов сильно уменьшается

### Не только поиск элемента

Бинарный поиск полезен и для "поиска по ответу", если условие монотонно:

- до некоторого значения ответ "нет"
- после него ответ "да"

Это более продвинутый, но очень важный паттерн.

## Как говорить о решении на собеседовании

Хороший порядок:

1. Сначала коротко пересказать задачу.
2. Назвать структуру данных.
3. Назвать идею решения.
4. Назвать сложность.
5. Потом писать код.

Если ты так делаешь, собеседующий видит мышление, а не просто печать символов.

## Типичные ошибки

- взять контейнер "по привычке"
- не подумать, нужен ли порядок
- писать вложенные циклы, когда достаточно словаря
- знать название паттерна, но не видеть условия его применимости
- не учитывать пустой вход и крайние случаи

## Что нужно вынести из лекции

- структура данных выбирается под операции, а не по красоте названия
- `set` нужен для уникальности, `map` для связи ключа со значением
- `unordered_*` удобно там, где важен быстрый доступ без порядка
- алгоритмические паттерны надо понимать через задачу и идею, а не через термин
