# Лекция 5. CMake, сборка и тесты: как проект перестает быть набором файлов

## Почему эта лекция важнее, чем кажется

Многие новички считают, что сборка - это второстепенная техническая деталь. На практике именно по сборке быстро видно:

- понимает ли человек проект как систему
- умеет ли он работать не только с кодом, но и с окружением
- сможет ли другой разработчик вообще запустить его работу

## Что такое сборка проекта

Сборка - это не одна команда компилятора, а целый процесс:

- найти исходные файлы
- передать компилятору правильные флаги
- подключить заголовки
- собрать объектные файлы
- слинковать итоговую программу
- при необходимости собрать тесты и библиотеки

Когда проект маленький, это можно сделать руками. Когда проект растет, ручной подход перестает быть надежным.

## Почему одной команды `g++ main.cpp` мало

Пока у тебя один файл, кажется, что все просто:

```bash
g++ -std=c++17 main.cpp -o app
```

Но сразу появляются вопросы:

- что если файлов несколько
- что если часть логики должна быть библиотекой
- что если у тебя тесты
- что если у проекта разные режимы сборки
- что если редактору нужен точный compile database

Именно для этого и нужна система сборки.

## Что такое CMake

CMake - это инструмент, который описывает, как должен собираться проект.

Важно понимать точно:

- CMake сам по себе не компилятор
- он генерирует конфигурацию для backend-инструмента вроде Ninja или Make

То есть он отвечает за "описание правил сборки", а не за машинный код напрямую.

## Минимальный пример

```cmake
cmake_minimum_required(VERSION 3.20)
project(task_tracker LANGUAGES CXX)

set(CMAKE_CXX_STANDARD 17)
set(CMAKE_CXX_STANDARD_REQUIRED ON)

add_executable(task_tracker src/main.cpp)
target_include_directories(task_tracker PRIVATE include)
```

Теперь разберем не формально, а по смыслу.

### `project(...)`

Объявляет проект и сообщает, какие языки участвуют.

### `CMAKE_CXX_STANDARD`

Говорит, какой стандарт языка требуется.

Это важно, потому что от стандарта зависит:

- доступный синтаксис
- поведение библиотеки
- совместимость с кодом

### `add_executable`

Говорит: из этого набора файлов надо собрать исполняемый файл.

### `target_include_directories`

Объясняет, где искать заголовочные файлы для данного target.

Это уже шаг от "магии include" к явной модели зависимостей.

## Что такое target и почему вокруг него строится современный CMake

Target - это единица сборки:

- приложение
- библиотека
- тестовая цель

Почему target важен:

- у каждой сущности свои зависимости
- свои include paths
- свои compile options

Если не мыслить target-ами, проект быстро превращается в глобальную смесь флагов и путей.

## Target-based подход

Плохой стиль:

- глобально прописать include directories и compile flags для всего подряд

Хороший стиль:

- связывать настройки с конкретной библиотекой или приложением

Это делает проект:

- понятнее
- локальнее
- устойчивее к росту

## Почему полезно делить проект на библиотеку и приложение

Пусть у тебя есть логика задач и CLI.

Если все лежит в `main.cpp`, получаются проблемы:

- трудно тестировать
- трудно переиспользовать
- точка входа становится перегруженной

Гораздо лучше:

```cmake
add_library(task_core
    src/task.cpp
    src/task_manager.cpp
)

target_include_directories(task_core PUBLIC include)

add_executable(task_cli src/main.cpp)
target_link_libraries(task_cli PRIVATE task_core)
```

Что это дает:

- `task_core` можно тестировать отдельно
- `main.cpp` становится тонким слоем
- проект начинает напоминать рабочую архитектуру

## Debug и Release

Эти режимы нужны не "для галочки".

### Debug

Нужен, когда:

- важна отладочная информация
- нужен понятный stack trace
- нужно пошагово проходить код

### Release

Нужен, когда:

- интересует более реальная производительность
- важны оптимизации

Важно помнить: иногда баг проявляется только в одном из режимов.

## Warning-ы

Почему warning-ы нельзя игнорировать:

- они часто указывают на подозрительное место
- позволяют увидеть потенциальную ошибку до рантайма

Примеры типовых warning-ов:

- неиспользуемая переменная
- подозрительное преобразование типов
- отсутствие возвращаемого значения
- потенциально неинициализированная переменная

На junior уровне привычка собирать код "чисто" - очень важный признак.

## Тесты

Тест нужен не потому, что “так делают взрослые”, а потому что код меняется. Тест позволяет быстро ответить:

- не сломал ли я старое поведение

### Unit test

Проверяет маленький кусок логики.

Пример:

- функция вычисления статистики
- парсер входной строки
- отдельный метод

### Integration test

Проверяет связку нескольких компонентов или все приложение целиком.

На твоем текущем уровне уже полезно хотя бы понимать разницу.

## `compile_commands.json`

Этот файл очень важен для `clangd`.

Он говорит редактору:

- какими флагами компилируется каждый файл
- где include directories
- как реально устроена сборка

Из-за этого LSP начинает:

- лучше видеть символы
- правильнее подсказывать типы
- меньше ошибаться в диагностике

## README как часть инженерии

README должен отвечать на простые вопросы:

- что это за проект
- как собрать
- как запустить
- как прогнать тесты

Если README нет, проект сильно хуже воспроизводится.

## Что нужно вынести из лекции

- сборка - часть проекта, а не второстепенная мелочь
- CMake описывает сборку, а не заменяет компилятор
- target-based подход делает зависимости и настройки явными
- тесты и README повышают устойчивость проекта и удобство работы с ним
