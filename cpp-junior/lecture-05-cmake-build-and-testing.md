# Лекция 5. Сборка проекта, CMake и тестирование без магии

## Почему это не "второстепенная инженерка"

Очень частая ошибка: считать, что "главное писать код, а сборка как-нибудь потом". На реальной работе это не так. Код, который нельзя стабильно собрать, протестировать и открыть в редакторе без боли, мало кому нужен.

## Что значит "собрать программу"

Когда проект маленький, ты можешь думать: "у меня есть `.cpp`, я его скомпилировал". Но как только файлов становится больше, начинается реальная инфраструктура проекта.

Сборка обычно включает:

- компиляцию исходных файлов
- линковку
- настройку флагов компилятора
- подключение библиотек
- сборку тестов

## Почему нельзя жить на ручной команде компилятора вечно

Для файла `main.cpp` можно написать что-то вроде:

```bash
g++ -std=c++17 main.cpp -o app
```

Но дальше появляются вопросы:

- а если файлов 10
- а если есть библиотека
- а если есть тесты
- а если нужен debug и release
- а если проект должен открываться у тебя и у другого человека одинаково

Именно здесь появляется система сборки.

## Что делает CMake

CMake - это инструмент описания сборки.

Он:

- не компилирует код сам по себе
- а генерирует конфигурацию для реального backend-инструмента, например Ninja или Make

То есть ты описываешь проект один раз, а дальше получаешь воспроизводимую сборку.

## Минимальный `CMakeLists.txt`

```cmake
cmake_minimum_required(VERSION 3.20)
project(task_tracker LANGUAGES CXX)

set(CMAKE_CXX_STANDARD 17)
set(CMAKE_CXX_STANDARD_REQUIRED ON)

add_executable(task_tracker src/main.cpp)
target_include_directories(task_tracker PRIVATE include)
```

Разберем по смыслу.

### `cmake_minimum_required`

Говорит, какая минимальная версия CMake нужна.

### `project(...)`

Объявляет проект.

### `CMAKE_CXX_STANDARD`

Указывает стандарт языка, например C++17.

### `add_executable`

Говорит: из этих исходников нужно собрать исполняемый файл.

### `target_include_directories`

Говорит, где искать заголовочные файлы для данного target.

## Что такое target

Это очень важное слово в CMake.

Target - это единица сборки:

- исполняемый файл
- библиотека
- иногда тестовая цель

Современный стиль CMake строится вокруг target.

Это важно потому, что настройки привязываются не ко "всему миру сразу", а к конкретной сущности:

- этому приложению
- этой библиотеке
- этим тестам

## Почему target-based подход лучше

Потому что он:

- уменьшает хаос
- делает зависимости явными
- легче масштабируется

Плохой стиль:

- глобально выставить все include paths и compile flags

Лучший стиль:

- явно связывать каждую настройку с конкретным target

## Библиотека и приложение

Очень полезно разделять:

- core logic
- CLI или точку входа

Пример:

```cmake
add_library(task_core
    src/task.cpp
    src/task_manager.cpp
)

target_include_directories(task_core PUBLIC include)

add_executable(task_cli src/main.cpp)
target_link_libraries(task_cli PRIVATE task_core)
```

Что это дает:

- логику можно тестировать отдельно
- `main.cpp` не превращается в центр всего проекта
- архитектура становится яснее

## Debug и Release

Это разные режимы сборки.

### Debug

Обычно:

- проще отлаживать
- больше отладочной информации
- меньше оптимизаций

### Release

Обычно:

- сильнее оптимизации
- меньше отладочной информации

Важно понимать: баг, который видно только в одном режиме, не редкость.

## Warning-ы

Предупреждения компилятора - это не "шум", который надо игнорировать. Часто это ранний сигнал, что в коде что-то подозрительное:

- неиспользуемая переменная
- неявное опасное преобразование
- потенциальная ошибка в логике

На junior уровне привычка читать и исправлять warning-ы очень полезна.

## Почему тесты вообще нужны

Тесты нужны не для галочки и не только для больших проектов.

Они дают:

- быструю проверку поведения
- защиту от случайной поломки при рефакторинге
- способ формализовать ожидания от кода

Если функция важна, полезно уметь быстро проверить:

- что она работает на обычных входах
- что она ведет себя адекватно на краях

## Unit tests

Это тесты маленьких частей логики.

Например:

- функция подсчета статистики
- парсер одной строки
- метод класса

Обычно unit test:

- маленький
- быстрый
- проверяет одну идею

## Что такое хороший маленький проект

На junior уровне уже хорошо, если структура выглядит так:

- `include/`
- `src/`
- `tests/`
- `CMakeLists.txt`
- `README.md`

Почему это лучше свалки файлов:

- код легче читать
- проще подключать новые модули
- проект легче объяснить

## `compile_commands.json`

Это важный практический файл для инструментов вроде `clangd`.

Если CMake запускается с:

```bash
cmake -S . -B build -G Ninja -DCMAKE_EXPORT_COMPILE_COMMANDS=ON
```

редактор получает точную информацию:

- какие include paths используются
- какие флаги компиляции заданы
- как реально собираются файлы

Без этого LSP в C++ часто работает заметно хуже.

## README как часть качества проекта

README нужен не потому, что "так принято", а потому что он отвечает на базовые вопросы другого человека:

- что это за проект
- как его собрать
- как его запустить
- как прогнать тесты

Если проект работает только у автора и только пока он сам сидит рядом, это плохой сигнал.

## Частые ошибки новичка

- все файлы в корне
- нет разделения на библиотеку и приложение
- сборка живет только в голове автора
- тестов нет
- warning-ы игнорируются
- CMake правят методом случайных проб

## Что нужно вынести из лекции

- сборка - это часть продукта, а не "вспомогательная мелочь"
- CMake нужен для воспроизводимой и масштабируемой сборки
- target-based подход делает проект понятнее
- тесты и README - это практические инструменты надежности и воспроизводимости
