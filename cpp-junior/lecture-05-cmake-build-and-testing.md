# Лекция 5. Сборка, CMake и тестирование

## Почему это важно

На рынке от junior C++ редко ждут "только код". Чаще ждут, что ты умеешь:

- собрать проект
- разложить его по файлам
- подключить тесты
- не бояться warning-ов

Если проект нельзя стабильно собрать, он мало чего стоит.

## Что делает CMake

CMake - это система описания сборки. Она не компилирует код сама по себе, а генерирует сборочную конфигурацию для инструмента вроде Ninja или Make.

Упрощенно:

1. ты описываешь проект в `CMakeLists.txt`
2. CMake генерирует build system
3. дальше выполняется сборка

## Минимальный проект

Структура:

- `include/`
- `src/`
- `tests/`
- `CMakeLists.txt`

Пример:

```cmake
cmake_minimum_required(VERSION 3.20)
project(task_tracker LANGUAGES CXX)

set(CMAKE_CXX_STANDARD 17)
set(CMAKE_CXX_STANDARD_REQUIRED ON)

add_executable(task_tracker
    src/main.cpp
    src/task_manager.cpp
)

target_include_directories(task_tracker PRIVATE include)
```

## Почему важен target-based подход

Хорошая практика в CMake:

- настраивать свойства на конкретных target
- не раздавать глобальные настройки без причины

То есть думать не "весь проект как масса файлов", а "вот приложение, вот библиотека, вот тесты".

## Библиотека + приложение

Часто хороший учебный проект выглядит так:

```cmake
add_library(task_core
    src/task.cpp
    src/task_manager.cpp
)

target_include_directories(task_core PUBLIC include)

add_executable(task_cli src/main.cpp)
target_link_libraries(task_cli PRIVATE task_core)
```

Это уже похоже на рабочую структуру.

## Warning-ы

Нужно привыкнуть включать строгие предупреждения.

Идея:

- компилятор подсказывает подозрительные места
- предупреждения лучше устранять рано

На junior уровне это сигнал аккуратности.

## Что такое тесты

Тест - это проверка ожидаемого поведения кода.

Минимум:

- функция получает вход
- проверяем ожидаемый выход

Пример идеи:

- `sum({1, 2, 3}) == 6`
- пустой список не ломает программу
- невалидный ввод обрабатывается явно

## Unit tests

Проверяют отдельную функцию или маленький модуль.

Это особенно полезно, когда:

- логика начинает разрастаться
- поведение надо сохранять при рефакторинге

## README как часть инженерной культуры

Даже маленький проект должен иметь:

- что это за проект
- как собрать
- как запустить
- как прогнать тесты

Это не бюрократия. Это способ сделать проект воспроизводимым.

## `compile_commands.json`

Для `clangd` очень полезно генерировать:

```text
cmake -S . -B build -G Ninja -DCMAKE_EXPORT_COMPILE_COMMANDS=ON
```

Тогда редактор лучше понимает проект.

## Типичные ошибки junior

- все файлы лежат в корне
- тестов нет
- CMake сломан при малейшем изменении
- include paths настроены случайно
- запуск проекта не описан

## Что считается хорошим результатом

- проект собирается одной-двумя командами
- структура понятна без объяснений
- тесты запускаются отдельно
- `clangd` и форматирование работают
