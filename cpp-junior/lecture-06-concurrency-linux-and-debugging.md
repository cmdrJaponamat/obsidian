# Лекция 6. Потоки, Linux-среда и как искать ошибки

## Почему эта лекция нужна junior

Даже если ты не идешь в hardcore low-level, от junior C++ часто ожидают:

- что он не путает процесс и поток
- что он умеет работать в Linux
- что он может хотя бы базово отлаживать программу

Не требуется быть экспертом по многопоточности. Требуется не быть беспомощным.

## Процесс и поток

### Процесс

Процесс - это запущенный экземпляр программы со своим адресным пространством и системными ресурсами.

### Поток

Поток - это отдельная линия выполнения внутри процесса.

Важно:

- несколько потоков одного процесса делят память процесса
- именно поэтому многопоточность мощная, но опасная

## Почему потоки вообще нужны

Потоки полезны, когда:

- нужно параллельно выполнять независимую работу
- нужно не блокировать приложение одной длинной операцией
- нужно обрабатывать несколько задач одновременно

Но потоками не надо лечить все подряд. Однопоточное корректное решение почти всегда лучше сырого "параллельного" хаоса.

## `std::thread`

Минимальный пример:

```cpp
#include <thread>
#include <iostream>

void worker() {
    std::cout << "Working\n";
}

int main() {
    std::thread t(worker);
    t.join();
}
```

### Что делает `join`

`join()` означает: дождаться завершения потока.

Если поток создан, но ты не синхронизировал его завершение корректно, программа может вести себя неправильно.

## Общие данные и race condition

Проблема появляется, когда два потока работают с одними и теми же данными.

```cpp
int counter = 0;
```

Если два потока одновременно делают `++counter`, итог может быть некорректным.

Это и есть одна из форм race condition: результат зависит от того, как именно по времени перемешались операции потоков.

## Почему `++counter` не "просто одна операция"

На уровне исходного кода выглядит просто, но в реальности это обычно не магический неделимый акт, а последовательность действий:

- прочитать значение
- изменить
- записать обратно

Если два потока вмешиваются друг в друга в этот момент, результат ломается.

## `std::mutex`

Mutex нужен, чтобы защитить критическую секцию, то есть участок кода, который должен выполняться только одним потоком за раз.

```cpp
std::mutex mutex;
```

## `std::lock_guard`

```cpp
void safe_increment() {
    std::lock_guard<std::mutex> lock(mutex);
    ++counter;
}
```

### Что это дает

- mutex захватывается при создании `lock`
- автоматически освобождается при выходе из области видимости

Это снова RAII. Именно поэтому `lock_guard` лучше ручного:

```cpp
mutex.lock();
// ...
mutex.unlock();
```

Ручной вариант опаснее:

- можно забыть `unlock`
- можно выйти раньше из функции
- можно получить исключение и не освободить mutex

## Deadlock

Deadlock - это ситуация, когда потоки ждут друг друга бесконечно.

Например, если один поток держит ресурс A и ждет B, а второй держит B и ждет A.

На твоем уровне важно:

- знать, что это реальная проблема
- не усложнять схему блокировок без нужды
- держать критические секции короткими

## Linux как рабочая среда

Для C++ под Linux нужно уверенно делать обычные вещи:

- искать файлы
- запускать сборку
- читать логи
- смотреть процессы
- останавливать процессы

Базовый набор команд:

- `pwd`
- `ls`
- `cd`
- `find`
- `rg`
- `ps`
- `kill`
- `less`
- `tail`

Если без GUI ты теряешься, это надо исправлять как можно раньше.

## Что такое отладка

Отладка - это не "смотреть на код и надеяться догадаться". Это процесс получения фактов о поведении программы.

Когда программа падает или работает странно, надо выяснить:

- где именно это происходит
- при каких входных данных
- какие значения были у переменных
- какой путь выполнения к этому привел

## Debug build

Чтобы удобно отлаживать, проект обычно собирают в debug-режиме:

- с отладочной информацией
- с меньшим уровнем оптимизаций

Иначе соответствие между исходным кодом и реальным выполнением хуже читается.

## Что такое stack trace

Stack trace показывает цепочку вызовов функций, которая привела к текущему месту выполнения или ошибке.

Это очень полезно, потому что помогает понять не только "где упало", но и "как программа сюда пришла".

## `gdb` на минимально рабочем уровне

На junior-уровне уже полезно уметь:

- запустить программу в отладчике
- поставить breakpoint
- выполнять код по шагам
- смотреть значения переменных
- получить backtrace

Не нужно знать весь `gdb` наизусть. Нужно перестать бояться его.

## Логирование

Иногда отладчик не лучший инструмент. Тогда помогают логи.

Но логировать надо осмысленно:

- входные данные
- ключевые шаги
- причины ошибок

Плохо:

- засорять код бессмысленными `cout`

Лучше:

- логировать важные факты

## Что нужно вынести из лекции

- многопоточность опасна не потому, что "сложный синтаксис", а потому что несколько потоков делят одни данные
- mutex и `lock_guard` защищают критические участки
- Linux CLI и отладка - часть рабочего набора junior C++
- при баге нужно собирать факты, а не гадать
