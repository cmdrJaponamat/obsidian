# Лекция 6. Потоки, Linux-практика и отладка

## Почему это рыночная тема

Для junior C++ под Linux очень часто ожидается хотя бы базовое понимание:

- потоков
- синхронизации
- командной строки
- отладки

Никто не требует от тебя сложных lock-free структур. Но требуется, чтобы ты не пугался процессов, логов и backtrace.

## Процесс и поток

### Процесс

Отдельный экземпляр программы со своим адресным пространством.

### Поток

Поток выполнения внутри процесса.

В одном процессе может быть несколько потоков, которые делят память процесса.

## `std::thread`

Пример:

```cpp
#include <thread>
#include <iostream>

void worker() {
    std::cout << "Work\n";
}

int main() {
    std::thread t(worker);
    t.join();
}
```

`join()` важен: он дожидается завершения потока.

## Race condition

Если несколько потоков обращаются к общим данным без корректной синхронизации, результат может зависеть от случайного порядка выполнения.

Пример проблемы:

```cpp
int counter = 0;
```

Если два потока одновременно увеличивают `counter`, итог может быть некорректным.

## `std::mutex` и `std::lock_guard`

Чтобы защитить общий ресурс:

```cpp
std::mutex mutex;

void safe_increment() {
    std::lock_guard<std::mutex> lock(mutex);
    // критическая секция
}
```

Почему `lock_guard` полезен:

- автоматически освобождает mutex
- лучше сочетается с RAII
- меньше риск забыть `unlock()`

## Deadlock

Это ситуация, когда потоки ждут друг друга бесконечно.

На junior уровне важно:

- понимать, что это бывает
- не писать сложные схемы блокировок без необходимости
- стараться держать критические секции короткими

## Linux CLI как часть повседневной работы

Тебе нужно быстро уметь:

- запускать сборку
- смотреть логи
- искать файлы и строки в проекте
- управлять процессами

Минимальный набор:

- `rg`
- `find`
- `ps`
- `kill`
- `tail`
- `less`

## Отладка

### Что делать при падении программы

1. собрать debug-версию
2. воспроизвести проблему
3. посмотреть stack trace
4. понять, где именно программа ломается

### Что такое stack trace

Это цепочка вызовов функций, которая привела к ошибке.

Это один из самых полезных инструментов диагностики.

## `gdb` на практическом уровне

Нужно уметь хотя бы:

- запустить программу под отладчиком
- поставить breakpoint
- пройти по шагам
- посмотреть значения переменных
- получить backtrace

Этого уже достаточно для очень многих junior-задач.

## Почему логирование важно

Не все проблемы удобно ловить отладчиком. Логи помогают:

- понять путь выполнения
- увидеть входные данные
- найти место странного поведения

Но логировать нужно осмысленно, а не превращать код в свалку `cout`.

## Практические правила

- сначала делай однопоточное корректное решение
- потом добавляй потоки
- если есть shared state, думай о синхронизации
- используй CLI как рабочий инструмент каждый день
- при баге не гадай, а собирай факты через отладку и логи
