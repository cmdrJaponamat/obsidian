# Лекция 6. Потоки, процессы, Linux и отладка: как программа живет вне исходного кода

## Почему эта лекция вообще нужна

До этого момента мы в основном говорили про код как текст и про структуры данных внутри одной логики. Но программа живет не на бумаге. Она:

- запускается как процесс
- может создавать потоки
- работает в окружении операционной системы
- пишет в файлы и консоль
- падает, зависает и иногда ведет себя странно

Junior-разработчик не обязан быть экспертом по ОС. Но он обязан понимать базовую модель исполнения программы.

## Процесс

Процесс - это запущенный экземпляр программы.

Когда ты запускаешь исполняемый файл, ОС создает процесс и выделяет ему:

- адресное пространство
- дескрипторы файлов
- другие системные ресурсы

Важно понимать:

- одна и та же программа может быть запущена как несколько независимых процессов
- у каждого процесса свое адресное пространство

Это значит, что два процесса не делят обычные переменные "просто так".

## Поток

Поток - это отдельная линия выполнения внутри одного процесса.

Ключевая разница с процессом:

- потоки одного процесса делят общую память процесса

Именно это делает потоки мощными и опасными одновременно.

## Почему потоки полезны

Потоки полезны, когда:

- несколько задач можно выполнять независимо
- одна задача долго ждет I/O
- нужно поддерживать отзывчивость приложения

Но это не значит, что многопоточность надо добавлять везде. Если однопоточное решение:

- проще
- корректно
- достаточно быстро

то оно часто лучше.

## `std::thread`

Минимальный пример:

```cpp
#include <iostream>
#include <thread>

void worker() {
    std::cout << "work\n";
}

int main() {
    std::thread t(worker);
    t.join();
}
```

### Что происходит здесь

- создается поток `t`
- в нем выполняется функция `worker`
- `join()` заставляет основной поток дождаться завершения `t`

Если не понять жизненный цикл потока, код быстро становится нестабильным.

## Почему `join()` важен

Поток - это ресурс. Его завершение должно быть обработано явно.

Если у тебя есть поток, который ты создал, но не синхронизировал его завершение корректно, можно получить:

- некорректное завершение программы
- потерю результата
- плохо определенное поведение

## Общие данные и race condition

Представим:

```cpp
int counter = 0;
```

И два потока одновременно делают:

```cpp
++counter;
```

Наивно кажется, что итог будет "плюс два". Но это не гарантировано.

## Почему `++counter` не атомарен в обычном смысле

На уровне исходного текста это короткая запись. Но на уровне выполнения это обычно последовательность шагов:

1. прочитать старое значение
2. вычислить новое
3. записать новое обратно

Если два потока вмешиваются друг в друга между этими шагами, итог может быть неверным.

Это и есть race condition:

- результат зависит от неуправляемого взаимного переплетения действий потоков

## `std::mutex`

`mutex` нужен, чтобы ограничить доступ к критической секции.

Критическая секция - это часть кода, где ты работаешь с общими данными и где одновременно должен находиться только один поток.

```cpp
std::mutex mutex;
```

## `std::lock_guard`

```cpp
void safe_increment() {
    std::lock_guard<std::mutex> lock(mutex);
    ++counter;
}
```

### Что дает `lock_guard`

- при создании он захватывает `mutex`
- при выходе из области видимости освобождает его

Это снова RAII: захват ресурса связан с временем жизни объекта.

## Почему `lock_guard` лучше ручного `lock/unlock`

Ручной код:

```cpp
mutex.lock();
// работа
mutex.unlock();
```

Проблемы:

- можно забыть `unlock`
- можно сделать ранний `return`
- можно бросить исключение и не дойти до освобождения

`lock_guard` делает этот шаблон безопаснее.

## Deadlock

Deadlock - это ситуация, когда потоки ждут друг друга бесконечно.

Типичная идея:

- поток A держит ресурс 1 и ждет ресурс 2
- поток B держит ресурс 2 и ждет ресурс 1

На junior уровне важно:

- понимать, что это реальный риск
- не строить сложные схемы блокировок без необходимости
- минимизировать время удержания блокировки

## Linux как рабочая среда

Для C++ под Linux нужно понимать не только язык, но и рабочую среду.

Минимум:

- как найти файл
- как найти строку в коде
- как запустить сборку
- как посмотреть процесс
- как остановить зависшую программу

## Почему CLI важен

Потому что именно в CLI ты быстро делаешь повседневные действия:

- `rg` для поиска
- `find` для файлов
- `ps` для процессов
- `kill` для остановки
- `less` и `tail` для логов

Если разработчик не умеет жить в командной строке, он теряет много времени на самые обычные операции.

## Что такое отладка

Отладка - это не "внимательно смотреть на код и надеяться понять". Это получение фактов о поведении программы.

Когда программа ломается, нужно ответить:

- где именно
- при каких условиях
- какие значения были у переменных
- как исполнение дошло до этого места

## Debug build

Отлаживать нужно обычно в debug-сборке.

Почему:

- есть отладочная информация
- связь между исходным кодом и исполнением понятнее
- оптимизации не искажают картину так сильно, как в release

## Stack trace

Stack trace показывает цепочку вызовов, которая привела программу к текущему месту.

Это очень важно, потому что ошибка может проявиться в одной функции, но прийти из совсем другой части программы.

## `gdb`

На твоем уровне уже полезно уметь:

- запускать программу в `gdb`
- ставить breakpoint
- смотреть переменные
- идти по шагам
- получать backtrace

Не нужно знать `gdb` целиком. Нужно перестать бояться его как "инструмента для старших разработчиков".

## Логирование

Отладчик не всегда лучший инструмент. Иногда полезнее собрать картину через логи.

Хороший лог отвечает на вопрос:

- что происходило перед ошибкой

Плохой лог:

- бессистемный поток `cout`

Хороший лог:

- фиксирует важные события
- отмечает значения, которые влияют на решение
- помогает восстановить ход исполнения

## Что нужно вынести из лекции

- процесс и поток - разные вещи
- потоки делят память процесса, и именно поэтому race condition реальна
- `mutex` и `lock_guard` нужны для контроля доступа к общим данным
- Linux CLI и отладка - часть рабочего набора разработчика, а не отдельная экзотика
