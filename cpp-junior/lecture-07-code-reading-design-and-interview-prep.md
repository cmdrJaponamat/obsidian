# Лекция 7. Чтение чужого кода, дизайн и как не провалиться на объяснении собственного решения

## Почему чтение кода - это отдельный навык

Новичок часто думает, что главное - уметь писать код. На практике большую часть рабочего времени разработчик:

- читает существующий код
- понимает, что уже сделано
- ищет место для изменения
- локализует баг

Поэтому умение читать код почти так же важно, как умение его писать.

## С чего начинать чтение незнакомого проекта

Плохой путь:

- открыть случайный большой файл
- утонуть в деталях

Хороший путь:

1. Найти точку входа.
2. Понять структуру каталогов.
3. Понять модель сборки.
4. Найти главные сущности.
5. Потом уже идти в детали.

Так ты строишь карту проекта, а не просто смотришь на отдельные деревья.

## Что искать в первую очередь

- `main`
- `CMakeLists.txt`
- основные библиотеки
- тесты
- классы и модули, через которые проходит основная логика

Это дает скелет понимания.

## Что такое плохой дизайн

Плохой дизайн - это не "код не по модной архитектуре". Это код, который трудно:

- понимать
- менять
- тестировать

## God object

God object - это класс, который делает почти все.

Например:

- читает ввод
- хранит состояние
- выполняет бизнес-логику
- пишет в файл
- печатает пользователю

Такой объект неудобен, потому что любая мелкая правка начинает затрагивать сразу много обязанностей.

## Смешение ответственности

Если одна функция одновременно:

- парсит вход
- валидирует
- считает
- пишет лог
- печатает результат

то с большой вероятностью ее будет тяжело поддерживать.

## Хорошее разделение ответственности

Нужно не "разрезать код на миллион файлов", а разделить роли:

- parsing
- core logic
- storage
- presentation / CLI

Так изменения в одной части меньше ломают другие.

## Интерфейс и реализация

Полезно различать:

- что модуль обещает делать
- как именно он это делает

Чем меньше внешний код знает о деталях реализации, тем легче менять внутренности модуля без лавины правок.

## Dependency injection без лишней философии

На простом уровне это означает:

- зависимости не создаются жестко внутри каждого компонента
- они могут быть переданы извне

Пример:

- сервис работает не с "жестко пришитым" файловым хранилищем
- а с абстракцией хранения, которую можно заменить

Это полезно для:

- тестирования
- смены реализации
- уменьшения связности

## Как рефакторить старый код

Полезная последовательность:

1. Понять текущее поведение.
2. Если возможно, зафиксировать его тестом.
3. Разделить длинные функции.
4. Убрать смешение ответственности.
5. Переименовать неясные сущности.

Ключевое правило:

- сначала понять, потом улучшать

## Что проверяют на junior-собеседовании

Обычно не “глубокую академическую архитектуру”, а:

- понимаешь ли ты базовые конструкции
- можешь ли объяснить свои решения
- умеешь ли назвать компромиссы
- можешь ли читать небольшой проект

## Как рассказывать о своем проекте

Хорошая структура ответа:

1. Что делает проект.
2. Какие у него основные модули.
3. Какие структуры данных выбраны.
4. Как организована сборка и тесты.
5. Какие ограничения и что улучшил бы дальше.

Такой рассказ показывает зрелость лучше, чем перечисление файлов или терминов.

## Что нужно вынести из лекции

- чтение проекта начинается с карты, а не со случайного файла
- плохой дизайн обычно виден по смешению ответственности и god object
- умение объяснить свой код - часть разработки, а не отдельный бонус
