# Лекция 8. Финальный проект, портфолио и когда уже можно идти откликаться

## Зачем вообще нужен финальный проект

До этого момента знания были разложены по темам:

- синтаксис
- память
- STL
- алгоритмы
- сборка
- отладка

Но реальная работа не приходит по кускам. Финальный проект нужен, чтобы собрать все в одну систему.

Он показывает:

- умеешь ли ты довести задачу до конца
- можешь ли держать структуру проекта в голове
- понимаешь ли, как связаны код, сборка, тесты и README

## Каким должен быть проект на junior-уровне

Он не должен быть огромным. Он должен быть:

- законченным
- понятным
- воспроизводимым
- объяснимым

Чаще всего этого уже достаточно.

## Что важнее: размер или завершенность

Завершенность.

Маленький проект, который:

- собирается
- тестируется
- имеет читаемую структуру
- понятен по README

значительно полезнее, чем большой недострой.

## Подходящие типы проектов

- `task-tracker-cli`
- `log-analyzer`
- `file-indexer`
- `mini-chat-server`, если уже хочешь тронуть сеть

Хороший признак проекта: его задачу можно объяснить одним-двумя предложениями.

## Что проект должен показать работодателю

Неявно он показывает:

- умеешь ли ты работать со структурой кода
- понимаешь ли lifetime и владение ресурсами
- умеешь ли пользоваться STL
- умеешь ли собирать проект через CMake
- умеешь ли оформить запуск и тестирование

То есть проект - это сжатая демонстрация инженерной адекватности.

## Как строить проект

Практичный порядок:

1. Выбрать минимальный scope.
2. Разложить задачу на модули.
3. Сделать рабочий happy path.
4. Добавить обработку ошибок.
5. Добавить тесты.
6. Привести в порядок README и структуру.

Очень важно не пытаться сразу сделать "идеальную архитектуру". Для junior это чаще всего путь к недострою.

## Что обычно ломает проект

- слишком большой замах
- отсутствие границ задачи
- отсутствие тестов
- хаос в структуре
- код работает только "на машине автора"
- автор не может объяснить, почему проект устроен именно так

## Как понять, что проект уже можно показывать

Хороший быстрый чеклист:

- проект собирается по README
- тесты запускаются
- код разбит на понятные части
- ключевые ошибки обрабатываются
- ты можешь объяснить каждый основной модуль

Если это выполнено, проект уже полезен для портфолио.

## Когда можно начинать откликаться

Откликаться не нужно только тогда, когда ты "знаешь вообще все". Такой момент не наступает.

Разумный минимум для первых откликов:

- понимаешь базовые типы, функции, ссылки и `const`
- понимаешь стек, кучу, RAII и `unique_ptr`
- используешь `vector`, `string`, `unordered_map`
- можешь решить типовые базовые задачи
- умеешь собрать проект через CMake
- имеешь хотя бы один законченный проект

Этого уже достаточно, чтобы начинать пробовать стажировки и junior-вакансии.

## Как говорить о слабых местах

На интервью не страшно признать, что ты чего-то еще не знаешь. Страшнее делать вид, что знаешь, и развалиться на уточняющих вопросах.

Лучше говорить так:

- "эту тему знаю на базовом уровне"
- "вот это использовал в проекте"
- "сюда пока не углублялся, но понимаю, зачем это нужно"

Это звучит гораздо взрослее, чем случайный набор терминов.

## Что нужно вынести из лекции

- финальный проект нужен не ради галочки, а как сборка всех навыков
- завершенность важнее размера
- junior готовность определяется не полнотой знаний, а способностью решать реальные небольшие задачи и объяснять решения

## Вопросы для самопроверки

1. Почему маленький завершенный проект лучше большого недостроя?
2. Какие свойства делают проект "показываемым"?
3. Что именно твой проект демонстрирует работодателю?
4. Готов ли ты уже сейчас к первым откликам?
