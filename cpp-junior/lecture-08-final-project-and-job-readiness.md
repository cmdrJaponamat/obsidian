# Лекция 8. Финальный проект и готовность к первой работе

## Почему проект важнее еще десяти мелких задач

Мелкие задачи нужны, чтобы набить руку. Но работодатель нанимает не "решателя пяти минутных упражнений", а человека, который может довести реальный кусок работы до законченного состояния.

Проект важен потому, что в нем одновременно проявляются:

- структура кода
- обработка ошибок
- сборка
- тесты
- объяснимость решений

## Каким должен быть проект junior-уровня

Он не должен быть огромным. Он должен быть:

- законченным
- понятным
- воспроизводимым
- объяснимым

Очень важно понять: маленький законченный проект намного ценнее большого недостроя.

## Что значит "законченным"

Проект закончен, если:

- он собирается с нуля по README
- в нем есть минимальный полезный сценарий
- есть понятная структура
- есть базовая обработка ошибок
- есть тесты хотя бы на критичную логику

## Какие проекты подходят

- `task-tracker-cli`
- `log-analyzer`
- `file-indexer`
- `student-records-manager`

Важно, чтобы задачу можно было ясно объяснить в одном-двух предложениях.

## Что проект показывает работодателю

Неявно проект отвечает на вопросы:

- умеешь ли ты работать со структурой кода
- понимаешь ли ты lifetime и владение
- умеешь ли пользоваться STL
- умеешь ли собирать проект
- умеешь ли описывать запуск и ограничения

## Как строить проект без самообмана

Хороший порядок:

1. Определить минимальный scope.
2. Разделить проект на модули.
3. Сделать работающий happy path.
4. Добавить обработку ошибок.
5. Добавить тесты.
6. Почистить README и структуру.

Плохой порядок:

- придумать сразу слишком большой замысел
- размазать усилия
- не закончить ничего

## Что чаще всего ломает проект новичка

- слишком большой scope
- отсутствие четкой границы задачи
- хаос в структуре
- отсутствие тестов
- код работает только у автора
- автор не может объяснить архитектурные решения

## Когда можно начинать откликаться

Когда у тебя есть:

- базовое понимание языка
- навык решать типовые задачи
- хотя бы один законченный проект
- умение объяснить этот проект

То есть не нужен “идеальный уровень”. Нужен рабочий и понятный минимум.

## Как говорить о слабых местах проекта

Нормально говорить:

- что пока не покрыто тестами
- какие ограничения у проекта есть
- что ты бы улучшил следующим шагом

Это гораздо лучше, чем делать вид, что проект идеален.

## Что нужно вынести из лекции

- проект нужен как интеграция всех навыков
- завершенность важнее размера
- готовность к первой работе определяется не полнотой знаний, а способностью делать и объяснять рабочие небольшие решения

## Вопросы для самопроверки

1. Почему маленький законченный проект лучше большого недостроя?
2. Что должен показывать твой проект работодателю?
3. Какие ограничения твоего проекта ты уже знаешь?
4. Можешь ли ты объяснить свой проект без ухода в случайные детали?
