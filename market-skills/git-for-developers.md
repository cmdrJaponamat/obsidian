# Курс: Git для разработчика

## Почему Git важен не как “еще одна утилита”, а как часть профессии

Git нужен не просто для хранения кода. Он нужен, чтобы:

- фиксировать изменения
- понимать историю проекта
- работать в команде
- разбирать, кто и когда изменил поведение

Для junior это один из самых заметных маркеров инженерной зрелости. Если код есть, но человек не умеет им управлять как историей изменений, это быстро становится видно.

## Что такое репозиторий

Репозиторий - это каталог проекта вместе с историей изменений.

Когда ты делаешь:

```bash
git init
```

Git начинает хранить:

- состояние файлов
- коммиты
- ссылки на ветки

То есть проект превращается не просто в набор файлов, а в последовательность зафиксированных состояний.

## Базовый цикл работы

Очень важно понять именно цикл, а не список команд:

1. Ты меняешь файлы.
2. Смотришь, что изменилось.
3. Выбираешь, что именно войдет в коммит.
4. Делаешь коммит с осмысленным сообщением.

## `git status`

Это команда, которую надо любить.

Она отвечает на вопрос:

- что сейчас изменено
- что уже в staging
- что еще не добавлено

Для junior привычка часто смотреть `git status` очень полезна, потому что она снижает хаос.

## Working tree и staging area

Это один из самых важных концептов Git.

### Working tree

Текущее состояние файлов на диске.

### Staging area

Промежуточная область, где ты собираешь, что именно войдет в следующий коммит.

Это важно потому, что коммитится не “все подряд автоматически”, а то, что ты явно подготовил.

## `git add`

`git add` добавляет изменения в staging area.

Это не “сохрани в репозиторий”, а:

- подготовь эти изменения к следующему коммиту

## `git commit`

Коммит - это зафиксированный снимок проекта плюс сообщение, объясняющее смысл изменений.

Хороший коммит:

- небольшой
- логически цельный
- с понятным сообщением

Плохой коммит:

- смешивает 5 разных тем
- называется `fix`, `update`, `asdf`

## Почему история коммитов важна

История коммитов - это:

- средство отладки
- объяснение эволюции проекта
- инструмент ревью

По хорошей истории видно:

- что человек делал
- как делил работу
- умеет ли он думать изменениями, а не хаотичным состоянием папки

## Ветки

Ветка - это отдельная линия развития истории.

Практический смысл:

- ты работаешь над фичей отдельно
- не ломаешь основную линию разработки

Для junior важно хотя бы уверенно понимать:

- создать ветку
- перейти в ветку
- слить ветку обратно

## `merge` и `rebase`

### `merge`

Объединяет две линии истории, сохраняя факт их отдельного существования.

### `rebase`

Переписывает историю так, как будто твои коммиты делались поверх другой ветки с самого начала.

На старте важнее не делать агрессивный rebase повсюду, а понимать:

- merge сохраняет реальную форму истории
- rebase делает историю более линейной, но может быть опаснее при неаккуратном использовании

## Конфликты

Конфликт возникает, когда Git не может автоматически решить, какая версия изменений правильная.

Это не катастрофа, а нормальная часть совместной работы.

Важно:

- понять, какие куски кода конфликтуют
- осознанно выбрать правильный итог
- не коммитить конфликт “лишь бы исчезли метки”

## `git diff`

Это твой основной инструмент перед коммитом.

Он показывает:

- что именно ты изменил

Очень полезная привычка:

- перед коммитом смотреть diff

Тогда в репозиторий реже попадают:

- случайные debug prints
- мусорные правки
- лишние файлы

## `.gitignore`

`.gitignore` нужен, чтобы не тащить в репозиторий:

- бинарники
- build-каталоги
- временные файлы
- IDE-мусор

Это не косметика. Если `.gitignore` плохой, репозиторий быстро загрязняется.

## Что нужно вынести из курса

- Git - это управление историей изменений, а не просто “залить код”
- staging area - центральная часть нормального рабочего процесса
- хорошие коммиты маленькие и осмысленные
- ветки и конфликты - обычная часть командной разработки

## Практика

1. Создай учебный репозиторий.
2. Сделай 10 осмысленных коммитов вместо одного большого.
3. Создай feature branch, внеси изменение и влей обратно.
4. Смоделируй merge conflict и реши его.
5. Настрой `.gitignore` для C++ и Rust проектов.
6. Найди конкретную строку в истории через `git blame` и разберись, что менял тот коммит.
