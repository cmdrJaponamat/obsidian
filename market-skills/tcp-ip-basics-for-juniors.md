# Курс: TCP/IP база для junior

## Почему сети нужны даже не сетевому разработчику

Даже если ты не пишешь полноценный сетевой сервис, на практике полезно понимать:

- что такое адрес и порт
- как клиент говорит с сервером
- чем TCP отличается от UDP
- почему HTTP не равен “вся сеть”

Без этого сетевой код и сетевые ошибки воспринимаются как магия.

## IP-адрес и порт

Очень грубо:

- IP-адрес определяет узел в сети
- порт определяет конкретную службу на этом узле

То есть адрес отвечает на вопрос “куда”, а порт - “кому именно на этом узле”.

## Клиент и сервер

Клиент:

- инициирует запрос

Сервер:

- принимает запросы и отвечает на них

Это базовая модель очень многих сетевых взаимодействий.

## TCP и UDP

### TCP

Протокол с установлением соединения и надежной доставкой.

Интуитивный смысл:

- данные должны прийти в правильном порядке
- потерянные пакеты переотправляются

### UDP

Более простой протокол без встроенной надежной доставки.

Интуитивный смысл:

- отправил пакет
- дальше гарантий меньше

Почему это бывает полезно:

- ниже накладные расходы
- подходит для сценариев, где скорость важнее полной надежности

## Что такое сокет

Сокет - это программный интерфейс для сетевого взаимодействия.

Практически:

- через сокет программа отправляет и принимает данные по сети

## Где тут HTTP

HTTP - это прикладной протокол, который обычно работает поверх TCP.

Это важно понимать, потому что новички часто путают уровни:

- TCP - транспорт
- HTTP - прикладной протокол обмена запросами и ответами

## DNS

DNS нужен для преобразования доменного имени в IP-адрес.

Это одна из причин, почему сетевой запрос может не получиться даже до этапа “соединение установлено”.

## Почему таймауты важны

Сеть может:

- отвечать долго
- не отвечать вовсе
- быть нестабильной

Поэтому таймауты и retries - не “излишняя перестраховка”, а нормальная часть сетевой инженерии.

## Что нужно вынести из курса

- IP и порт играют разные роли
- TCP и UDP решают разные задачи
- HTTP - это не синоним всей сети
- сетевые ошибки естественны и требуют нормальной обработки

## Практика

1. Напиши минимальный TCP echo client/server.
2. Посмотри открытые порты в системе.
3. Сделай HTTP-клиент, который получает и разбирает ответ.
4. Смоделируй сетевую ошибку и обработай ее.
