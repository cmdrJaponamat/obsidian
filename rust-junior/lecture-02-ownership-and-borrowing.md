# Лекция 2. Ownership, borrowing и почему Rust не дает жить "как попало"

## Почему это центральная тема

Если не понять ownership, Rust будет казаться набором странных запретов. Если понять, то большая часть языка начнет складываться в одну систему.

Ownership - это способ явно описывать:

- кто владеет данными
- кто только временно ими пользуется
- когда данные уничтожаются

## Что значит "владеть значением"

Упрощенно:

- у значения есть владелец
- пока владелец жив, значение существует
- когда владелец выходит из области видимости, значение уничтожается

Пример:

```rust
{
    let text = String::from("hello");
} // здесь text уничтожается
```

Это очень похоже на идею lifetime объектов в C++, но в Rust модель владения выражена особенно строго и последовательно.

## Почему это важно

Без строгой модели владения в языках без GC часто появляются проблемы:

- use-after-free
- двойное освобождение
- неясное владение ресурсом
- data races

Rust пытается не допустить эти ситуации на уровне компиляции.

## Move semantics

Для многих типов в Rust присваивание означает не копию, а перемещение владения.

```rust
let first = String::from("hello");
let second = first;
```

После этого `first` использовать нельзя.

## Почему `first` больше нельзя использовать

Потому что `String` владеет своими данными. Если бы после присваивания оба имени считались полноправными владельцами, возник бы вопрос:

- кто должен освободить память

Rust решает это жестко:

- владение перешло к `second`
- `first` больше не владелец

Это кажется неудобным только до тех пор, пока не начинаешь видеть, сколько ошибок эта модель предотвращает.

## Почему некоторые типы копируются

Типы вроде `i32` обычно копируются, а не перемещаются в практическом смысле работы с кодом:

```rust
let a = 5;
let b = a;
```

Здесь `a` остается доступным.

Причина в том, что для маленьких простых типов копирование дешево и безопасно. Для `String` ситуация другая: там есть владение динамической памятью.

## Borrowing

Если ты не хочешь передавать владение, можно дать ссылку.

### Immutable borrow

```rust
fn print_text(text: &String) {
    println!("{text}");
}
```

Здесь функция не владеет строкой, а только временно берет ее "почитать".

### Mutable borrow

```rust
fn append_mark(text: &mut String) {
    text.push('!');
}
```

Здесь функция получает временный изменяемый доступ.

## Главное правило ссылок

В один момент времени:

- либо много неизменяемых ссылок
- либо одна изменяемая ссылка

Это правило не случайно. Оно защищает от очень неприятного класса проблем: конкурентного или неявного изменения данных в момент чтения.

## Что означает immutable borrow

Если у тебя есть:

```rust
let text = String::from("hello");
let r1 = &text;
let r2 = &text;
```

это нормально: несколько частей кода могут безопасно читать одно и то же значение.

## Что означает mutable borrow

Если у тебя есть:

```rust
let mut text = String::from("hello");
let r = &mut text;
```

значит, в этот момент Rust требует эксклюзивность:

- никто другой не должен одновременно читать или менять этот объект

Почему:

- изменение и чтение одновременно создают риск неясного состояния

## Почему borrow checker ругается

Не потому, что "Rust вредный". Он проверяет инвариант:

- не существует опасной комбинации доступа к данным

Это особенно важно потому, что Rust старается делать безопасность не договоренностью, а частью модели программы.

## `String`, `&String` и `&str`

Эту тему нужно понимать очень четко.

### `String`

Владеет строковыми данными.

### `&String`

Ссылка на `String`.

### `&str`

Строковый срез. Более общий способ сказать: "вот view на строковые данные".

## Почему `&str` часто лучше в параметрах функции

Если функция только читает строку, сигнатура:

```rust
fn print_name(name: &str)
```

обычно лучше, чем:

```rust
fn print_name(name: &String)
```

Почему:

- `&str` более общий интерфейс
- функция может принимать и строковый литерал, и ссылку на `String`

То есть такой интерфейс гибче и честнее по смыслу: мне не нужен именно `String`, мне нужен доступ к строковым данным.

## Почему `.clone()` - не универсальное решение

Очень частая ошибка новичка:

- увидел ошибку borrow checker
- добавил `.clone()`
- компилятор замолчал

Но проблема не исчезла, а была замазана.

Что плохого в бессмысленных `clone()`:

- лишние копии
- лишние затраты памяти и времени
- потеря ясности модели владения

Сначала надо ответить на вопрос:

- функция должна владеть значением или нет

## Как правильно думать о сигнатуре функции

Если функция должна забрать данные себе:

```rust
fn consume(text: String)
```

Если функция только читает:

```rust
fn read(text: &str)
```

Если функция должна изменить значение:

```rust
fn modify(text: &mut String)
```

Это и есть выражение модели данных через интерфейс.

## Область видимости и ссылки

Ссылка не может жить дольше, чем объект, на который она указывает.

Это один из самых важных инвариантов Rust.

Именно из-за него компилятор запрещает сценарии, которые в других языках могли бы привести к висячим ссылкам.

## Что нужно вынести из лекции

- ownership описывает, кто отвечает за lifetime данных
- присваивание для владеющих типов часто означает перенос владения
- borrowing нужен, когда ownership передавать не надо
- `&str` часто лучше `&String` для чтения строки
- `clone()` нельзя использовать как универсальную заплатку
