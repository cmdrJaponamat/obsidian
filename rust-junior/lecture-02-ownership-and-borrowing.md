# Лекция 2. Ownership и borrowing

## Почему это сердце Rust

Если не понять ownership, язык будет казаться набором запретов. Если понять - станет ясно, что компилятор просто заставляет тебя явно выражать модель владения данными.

## Основная идея ownership

Упрощенная модель:

- у каждого значения есть владелец
- в один момент времени владелец отвечает за lifetime этого значения
- когда владелец выходит из области видимости, значение уничтожается

Пример:

```rust
{
    let text = String::from("hello");
} // здесь text уничтожается
```

## Move semantics

В Rust многие типы по умолчанию перемещаются, а не копируются.

```rust
let first = String::from("hello");
let second = first;
```

После этого `first` больше использовать нельзя, потому что владение перешло к `second`.

Это защищает от ситуации, где несколько владельцев случайно думают, что каждый отвечает за одну и ту же память.

## Borrowing

Если владение передавать не нужно, можно брать ссылку.

### Immutable borrow

```rust
fn print_text(text: &String) {
    println!("{text}");
}
```

### Mutable borrow

```rust
fn append_mark(text: &mut String) {
    text.push('!');
}
```

## Главное правило ссылок

В один момент времени:

- либо много immutable borrow
- либо один mutable borrow

Это один из ключевых инвариантов Rust.

## Почему так сделано

Чтобы компилятор мог предотвратить:

- data races
- неконтролируемое изменение данных
- использование невалидных ссылок

## `String`, `&String` и `&str`

Практический смысл:

- `String` - владеющая строка
- `&String` - ссылка на `String`
- `&str` - строковый срез, более общий вид ссылки на строковые данные

Во многих интерфейсах лучше принимать `&str`, если нужно просто читать строку.

```rust
fn print_name(name: &str) {
    println!("{name}");
}
```

Это обычно гибче, чем `&String`.

## Почему не надо лечить все через `.clone()`

Новичок часто видит ошибку borrow checker и делает `clone()`, чтобы "отстало".

Проблема:

- лишние копии
- потеря понимания модели данных
- код становится тяжелее и хуже

Сначала нужно понять:

- кто должен владеть значением
- кому достаточно ссылки

## Пример нормального мышления

Вопрос:

- функция должна владеть строкой после вызова?

Если да:

```rust
fn consume(text: String) { ... }
```

Если нет:

```rust
fn read(text: &str) { ... }
```

Если должна изменить строку:

```rust
fn change(text: &mut String) { ... }
```

## Область видимости и ссылки

Ссылка не должна жить дольше объекта, на который она указывает.

Именно поэтому компилятор запрещает ряд потенциально опасных сценариев.

## Что на самом деле делает borrow checker

Он не "вредничает". Он проверяет инвариант:

- ни одна ссылка не указывает на уже недействительные данные
- параллельное изменение и чтение не создают неконтролируемые состояния

## Как учиться проходить ошибки borrow checker

Полезный порядок:

1. Прочитать сообщение компилятора полностью.
2. Понять, где создается владелец.
3. Понять, кто и когда берет borrow.
4. Переписать модель данных, а не засыпать код `clone()`.

## Типичные ошибки новичка

- бессмысленные `clone()`
- передача владения там, где нужна ссылка
- mutable borrow слишком широкой области
- попытка одновременно читать и мутировать одну структуру
