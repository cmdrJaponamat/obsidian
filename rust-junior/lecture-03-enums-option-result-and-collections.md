# Лекция 3. Enum, Option, Result и явное моделирование состояний

## В чем сила Rust на уровне прикладного кода

Rust заставляет гораздо честнее описывать возможные состояния программы.

Во многих кодовых базах ошибки выглядят так:

- функция может не найти результат, но это как-то неочевидно
- функция может упасть, но сигнатура об этом не говорит
- разные варианты состояния спрятаны в строках и флагах

Rust подталкивает к другому стилю:

- отсутствие значения выражается типом
- ошибка выражается типом
- набор вариантов состояния выражается типом

## `enum`

`enum` описывает конечный набор вариантов.

```rust
enum Command {
    Add(String),
    List,
    Done(usize),
}
```

Здесь команда - не "какая-то строка", а строго типизированное состояние.

## Почему это лучше строки

Если ты передаешь команду как строку:

- легко ошибиться в значении
- логика разбросана по условным конструкциям
- часть состояний не описана явно

Если ты используешь `enum`:

- варианты перечислены в одном месте
- компилятор помогает обработать их явно

## `Option`

`Option<T>` означает:

- либо значение есть: `Some(T)`
- либо значения нет: `None`

Пример:

```rust
fn find_name(id: u32) -> Option<String> {
    if id == 1 {
        Some(String::from("Ivan"))
    } else {
        None
    }
}
```

## Почему `Option` лучше неявного `null`

Потому что отсутствие значения больше не скрыто.

Если функция возвращает `Option<String>`, любой читатель кода сразу понимает:

- значение может отсутствовать
- это надо обработать

Это гораздо честнее, чем "может быть пустая строка", "может быть специальный код", "может быть что-то еще".

## `Result`

`Result<T, E>` означает:

- `Ok(T)` - операция успешна
- `Err(E)` - операция завершилась ошибкой

Пример:

```rust
fn parse_number(text: &str) -> Result<i32, std::num::ParseIntError> {
    text.parse::<i32>()
}
```

## Почему `Result` так важен

Он делает ошибку частью интерфейса функции.

Когда ты видишь:

```rust
fn load() -> Result<Data, Error>
```

ты сразу понимаешь:

- загрузка может не получиться
- вызывающий код обязан это учесть

Это очень полезная дисциплина.

## `match`

Работа с `Option` и `Result` очень часто идет через `match`.

```rust
match parse_number("42") {
    Ok(value) => println!("parsed: {value}"),
    Err(error) => println!("error: {error}"),
}
```

### Почему `match` важен

Он заставляет явно разбирать варианты состояния.

Именно поэтому Rust так хорошо подталкивает к "честному" стилю обработки данных.

## `if let`

Когда тебя интересует только один вариант, удобно использовать `if let`.

```rust
if let Some(name) = find_name(1) {
    println!("{name}");
}
```

Это короче, чем полный `match`, если остальные варианты тебе не нужны.

## `unwrap()` и почему с ним осторожно

`unwrap()` говорит:

- "я уверен, что здесь есть значение"
- или "я уверен, что ошибки нет"

Если уверенность ложная, программа завершится panic.

На старте `unwrap()` может быть удобен в маленьких экспериментах, но в реальном коде им нельзя злоупотреблять.

Почему:

- ты теряешь контролируемую обработку ошибок
- программа падает там, где могла бы объяснить проблему пользователю

## Коллекции

### `Vec<T>`

Это основной динамический массив.

```rust
let mut values = vec![1, 2, 3];
values.push(4);
```

Подходит, когда:

- нужен список элементов
- важен порядок
- нужен проход по всем элементам

### `HashMap<K, V>`

Это словарь.

```rust
use std::collections::HashMap;

let mut counts = HashMap::new();
counts.insert(String::from("rust"), 1);
```

Подходит, когда:

- нужен быстрый доступ по ключу
- нужно считать частоты
- нужно индексировать данные

## Итераторы на базовом уровне

В Rust ты часто будешь работать не через ручные индексы, а через итераторы.

Пример:

```rust
let values = vec![1, 2, 3, 4];
let sum: i32 = values.iter().sum();
```

На старте достаточно понимать:

- `iter()` - читать элементы по ссылке
- `iter_mut()` - получать изменяемый доступ
- `into_iter()` - часто передавать владение элементами дальше

Позже это станет важнее в деталях, особенно вместе с ownership.

## Что нужно вынести из лекции

- `enum` помогает явно описывать варианты состояния
- `Option` делает отсутствие значения честной частью интерфейса
- `Result` делает ошибку честной частью интерфейса
- `match` и `if let` - базовые инструменты работы с такими типами
- `Vec` и `HashMap` закрывают очень много прикладных задач
