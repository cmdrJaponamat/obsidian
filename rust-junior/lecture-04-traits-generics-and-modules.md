# Лекция 4. Traits, generics и модули

## Зачем это нужно

После первых задач на одном файле надо перейти к нормальной структуре проекта. Для этого нужно понять:

- как делить код на модули
- как выражать поведение через `trait`
- как не дублировать код через generics там, где это уместно

## `struct` и `impl`

Базовая модель данных:

```rust
struct Task {
    title: String,
    done: bool,
}
```

Логику добавляют через `impl`:

```rust
impl Task {
    fn mark_done(&mut self) {
        self.done = true;
    }
}
```

## Что такое `trait`

`trait` описывает набор поведения.

```rust
trait Storage {
    fn save(&self, text: &str);
}
```

Это похоже на интерфейс в других языках, но у Rust есть своя модель использования и композиции через traits.

## Зачем traits нужны practically

- описывать общий контракт
- подменять реализации
- строить более гибкую архитектуру
- писать обобщенный код

## Generics

Generics позволяют писать код, работающий с разными типами.

```rust
fn print_value<T: std::fmt::Display>(value: T) {
    println!("{value}");
}
```

На junior уровне важно не злоупотреблять generics. Используй их там, где они реально упрощают и убирают дублирование.

## Ограничения через bounds

Пример:

```rust
fn compare_and_print<T: Ord + std::fmt::Display>(a: T, b: T) {
    if a > b {
        println!("{a}");
    } else {
        println!("{b}");
    }
}
```

Это говорит: тип `T` должен уметь сравниваться и печататься.

## Модули

Модули помогают разложить проект на понятные части.

Пример структуры:

- `main.rs`
- `lib.rs`
- `parser.rs`
- `storage.rs`
- `domain.rs`

Это уже намного лучше, чем весь проект в одном файле.

## `pub`

По умолчанию элементы Rust приватные. Это полезно, потому что заставляет осознанно открывать только нужное.

```rust
pub struct Task {
    pub title: String,
}
```

Но не надо бездумно ставить `pub` везде. Лучше открывать только то, что действительно часть интерфейса.

## Хорошая структура маленького проекта

Например:

- `domain` - модели и инварианты
- `parser` - разбор CLI-команд
- `storage` - сохранение/загрузка
- `app` - координация логики

Это делает код понятнее и тестируемее.

## Trait для хранилища

Очень полезный учебный паттерн:

- `InMemoryStorage`
- `FileStorage`

Обе реализации соответствуют одному `trait`.

Это помогает понять:

- абстракцию
- границы ответственности
- как подменять зависимости

## Типичные ошибки новичка

- все в одном файле
- `pub` везде подряд
- сложные generics ради "красоты"
- trait там, где достаточно обычной функции или структуры
