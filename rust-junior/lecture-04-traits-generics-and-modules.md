# Лекция 4. Struct, trait, generics и как раскладывать Rust-проект по модулям

## Почему после базового синтаксиса начинается настоящая архитектура

Пока код маленький, можно терпеть один файл. Но как только появляется:

- несколько сущностей
- работа с файлами
- несколько сценариев использования
- тесты

нужна структура.

В Rust для этого особенно важны:

- `struct`
- `impl`
- `trait`
- модули
- generics

## `struct`

`struct` описывает данные.

```rust
struct Task {
    title: String,
    done: bool,
}
```

Это не "класс с магией". Это прежде всего форма хранения состояния.

## `impl`

Через `impl` к структуре добавляют методы.

```rust
impl Task {
    fn mark_done(&mut self) {
        self.done = true;
    }
}
```

Это уже напоминает объектную модель:

- есть состояние
- есть операции над этим состоянием

## Зачем это полезно

Когда логика привязана к данным:

- код становится более собранным
- инварианты проще поддерживать
- проект легче читать

## Что такое `trait`

`trait` описывает поведение, которое может поддерживаться разными типами.

```rust
trait Storage {
    fn save(&self, text: &str);
}
```

Практический смысл:

- это контракт
- разные типы могут реализовать его по-своему

## Почему `trait` полезен

Пример:

- есть in-memory storage
- есть file-based storage

Обе сущности делают похожую вещь: сохраняют данные. Но реализуют это по-разному.

Если выразить общий контракт через `trait`, код становится гибче.

## Generics

Generics позволяют писать код, который работает с разными типами.

```rust
fn print_value<T: std::fmt::Display>(value: T) {
    println!("{value}");
}
```

На старте важна не глубина обобщенного программирования, а понимание идеи:

- если логика одинакова для разных типов, ее можно обобщить

## Почему не надо злоупотреблять generics

Новичок иногда видит их как "признак продвинутости". На деле это может только усложнить код.

Используй generics, когда:

- реально убираешь дублирование
- интерфейс остается понятным

Если generic делает код хуже читаемым без реальной пользы, он не нужен.

## Bounds

Когда ты пишешь:

```rust
fn compare<T: Ord>(a: T, b: T) -> T
```

ты говоришь: тип `T` должен поддерживать операции сравнения порядка.

Это делает требования функции явными.

## Модули

Модули нужны, чтобы делить код на части по смыслу.

Типичная структура:

- `main.rs`
- `lib.rs`
- `domain.rs`
- `parser.rs`
- `storage.rs`

Это уже позволяет мыслить проектом, а не одним файлом.

## Что такое `pub`

По умолчанию элементы Rust приватные.

Это полезно, потому что заставляет явно решать:

- что является внутренней реализацией
- что является внешним интерфейсом

Если ставить `pub` везде подряд, структура быстро размывается.

## Хороший маленький проект

Например:

- `domain` - модели данных
- `parser` - разбор команд
- `storage` - сохранение и загрузка
- `app` - координация сценариев

Это здоровая структура для CLI-проекта junior-уровня.

## Trait для хранилища как учебный прием

Это очень полезный шаблон.

Можно сделать:

- `InMemoryStorage`
- `FileStorage`

и обе реализации привязать к одному `trait`.

Что это дает:

- код легче тестировать
- можно подменять реализацию
- архитектура становится более явной

## Что нужно вынести из лекции

- `struct` хранит состояние
- `impl` добавляет поведение
- `trait` выражает общий контракт
- модули помогают разложить проект по ответственности
- generics полезны, но только там, где реально улучшают код
