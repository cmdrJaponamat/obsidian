# Лекция 4. Traits, generics и модули: как проект перестает быть одним файлом

## Почему эта тема важна

После первых программ новички часто попадают в две крайности:

- держат весь код в одном файле
- или начинают строить сложную архитектуру, не понимая, зачем она нужна

Нужно идти средним путем:

- сначала понять, какие сущности вообще есть
- потом понять, какое поведение у них общее
- потом разложить код по модулям по смыслу

## `struct`

`struct` описывает форму данных.

```rust
struct Task {
    title: String,
    done: bool,
}
```

Это еще не "объектная система". Это просто способ сказать:

- у задачи есть такие-то поля

## `impl`

Через `impl` к структуре добавляют методы.

```rust
impl Task {
    fn mark_done(&mut self) {
        self.done = true;
    }
}
```

Здесь важна идея:

- данные и логика, которая естественно относится к этим данным, живут рядом

## Что такое `trait`

`trait` описывает общий контракт поведения.

```rust
trait Storage {
    fn save(&self, text: &str);
}
```

Это значит:

- любой тип, который реализует `Storage`, обещает иметь такой метод

## Почему `trait` полезен

Представь, что у тебя есть два способа хранения:

- в памяти
- в файле

Они разные по реализации, но у обоих может быть общий смысл:

- сохранить данные
- загрузить данные

`trait` позволяет выразить этот общий контракт.

## Почему `trait` не надо добавлять ради красоты

Если у тебя одна конкретная реализация и нет реальной абстракции, `trait` может быть лишним.

То есть вопрос не "можно ли сделать trait", а:

- есть ли здесь общий контракт, который реально нужен

## Generics

Generics позволяют писать код, который работает с разными типами.

```rust
fn print_value<T: std::fmt::Display>(value: T) {
    println!("{value}");
}
```

### Что это значит

Тип `T` здесь не фиксирован заранее, но он должен удовлетворять ограничению:

- уметь печататься через `Display`

## Почему generics полезны

Они позволяют:

- убрать дублирование
- писать более общий код
- не привязываться к одному конкретному типу без нужды

## Почему generics могут вредить

Если использовать их без причины:

- код становится сложнее читать
- сигнатуры перегружаются
- начинающий разработчик перестает понимать, где реальная польза

На твоем уровне важнее научиться видеть, когда generic действительно снимает дублирование, а когда это "архитектура ради архитектуры".

## Bounds

Когда ты пишешь:

```rust
fn compare<T: Ord>(a: T, b: T) -> T
```

ты говоришь:

- функция работает не с любым типом
- а только с тем, который умеет сравниваться по порядку

Это делает требования функции явными.

## Модули

Модуль - это способ группировать код по смыслу.

Простейшая причина использовать модули:

- не держать весь проект в одном файле

Пример структуры:

- `main.rs`
- `domain.rs`
- `parser.rs`
- `storage.rs`

Это уже сильный шаг вперед по сравнению с одной огромной точкой входа.

## Что такое `pub`

В Rust элементы по умолчанию приватные.

Это полезно, потому что заставляет явно решать:

- что является внутренней реализацией
- что является публичным интерфейсом

Если ставить `pub` везде, ты теряешь эту границу.

## Как выглядит здоровая структура маленького проекта

Например:

- `domain` - структуры данных и инварианты
- `parser` - разбор входа
- `storage` - работа с хранением
- `app` - координация логики

Это уже позволяет:

- читать проект блоками
- тестировать части отдельно
- не смешивать все ответственности в `main`

## Trait для storage как хороший учебный пример

Пусть у тебя есть:

- `InMemoryStorage`
- `FileStorage`

Если они оба реализуют `Storage`, ты получаешь:

- общий интерфейс
- возможность подмены реализации
- более ясную архитектуру

Это хорошее упражнение именно потому, что в нем абстракция не надумана.

## Что нужно вынести из лекции

- `struct` описывает данные
- `impl` добавляет поведение, связанное с этими данными
- `trait` нужен для общего контракта, а не для декоративной сложности
- generics полезны, когда реально устраняют дублирование
- модули помогают строить проект из смысловых частей
