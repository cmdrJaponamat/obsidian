# Лекция 5. Тесты, CLI, файлы и сериализация через serde

## Почему с этого места начинается "полезный код"

До этого момента можно было писать учебные функции. Но на практике junior Rust-задачи часто выглядят так:

- принять аргументы командной строки
- обработать данные
- прочитать или записать файл
- вернуть нормальную ошибку
- иметь хотя бы базовые тесты

Именно здесь код начинает походить на небольшое рабочее приложение.

## Зачем нужны тесты

Тест нужен, чтобы зафиксировать ожидаемое поведение.

Это важно потому, что код меняется:

- ты рефакторишь
- добавляешь функциональность
- исправляешь ошибку

Без тестов очень легко сломать уже работавшую часть программы.

## Unit tests

Unit test проверяет маленький кусок логики.

Пример:

- функция парсинга команды
- функция преобразования данных
- метод структуры

Минимальный пример:

```rust
fn add(a: i32, b: i32) -> i32 {
    a + b
}

#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn adds_numbers() {
        assert_eq!(add(2, 3), 5);
    }
}
```

### Что здесь важно

- `#[test]` отмечает тестовую функцию
- `assert_eq!` проверяет ожидаемое значение
- тесты лежат рядом с кодом, если это удобно

## Integration tests

Они проверяют уже не маленькую функцию, а работу проекта как целого.

Например:

- запустить CLI с аргументами
- проверить поведение готового сценария

Это особенно полезно, когда приложение уже начинает быть "настоящим", а не просто набором функций.

## CLI: что это на практике

CLI - это command-line interface, то есть программа, которую запускают из терминала с аргументами.

Например:

```bash
todo add "buy milk"
todo list
todo done 3
```

Такой формат очень полезен в обучении, потому что заставляет:

- четко определить команды
- разобрать аргументы
- обрабатывать ошибки ввода

## Почему CLI хорош для обучения

Он достаточно простой, но уже покрывает много реальных навыков:

- парсинг аргументов
- структура проекта
- хранение данных
- обработка ошибок
- тестирование

## Работа с файлами

Как только приложение должно сохранять состояние, появляется работа с файловой системой.

На этом этапе важно понять:

- чтение файла может не получиться
- запись файла может не получиться
- формат данных должен быть определен явно

То есть файловые операции - это не "побочный эффект", а полноценная часть модели ошибок.

## Почему файловые операции почти всегда должны возвращать `Result`

Потому что реально может произойти:

- файл не существует
- нет прав доступа
- формат содержимого неправильный
- диск недоступен

Если код это скрывает, приложение становится хрупким.

## Сериализация

Сериализация - это преобразование структуры данных в формат хранения или передачи:

- JSON
- TOML
- YAML
- бинарный формат

Для junior-приложений JSON - хороший старт:

- читается человеком
- легко дебажить
- хорошо поддержан экосистемой

## `serde`

`serde` - стандартный инструмент Rust для сериализации и десериализации.

Пример:

```rust
use serde::{Deserialize, Serialize};

#[derive(Serialize, Deserialize)]
struct Task {
    title: String,
    done: bool,
}
```

Что это дает:

- не нужно писать ручной парсинг JSON
- структура данных и формат хранения связаны типами

## Почему это удобно

Если ты сам пишешь сериализацию, легко ошибиться:

- забыть поле
- испортить формат
- не обработать крайние случаи

`serde` снимает много рутинной работы и дает надежную основу.

## Обработка ошибок в CLI

Плохой подход:

- все через `unwrap()`

Почему плохо:

- пользователь получает panic вместо внятного сообщения
- трудно понять, что произошло

Лучше:

- возвращать `Result`
- печатать понятное сообщение
- разделять пользовательские ошибки и внутренние ошибки программы

## Хороший маленький CLI-проект

Обычно это уже:

- понятные команды
- хранение состояния в файле
- хотя бы базовые тесты
- читаемый `README`
- нормальные сообщения об ошибках

Это выглядит как очень достойный junior-проект.

## Что нужно вынести из лекции

- тесты нужны для фиксации поведения, а не для галочки
- CLI-проект - отличный формат практики
- файловые операции всегда требуют честной обработки ошибок
- `serde` экономит массу рутины и делает код надежнее
