# Лекция 5. Тесты, CLI, файлы и serde: когда учебный код становится инструментом

## Почему эта тема важна

После базового синтаксиса и ownership нужно перейти к тому, что реально часто пишут на junior Rust:

- CLI-инструменты
- утилиты с файлами
- сериализация данных
- тестируемая логика

Здесь язык начинает работать не как набор конструкций, а как среда для построения законченных приложений.

## Тесты

Тест нужен, чтобы формально описать ожидаемое поведение кода.

Если код меняется, тест отвечает на вопрос:

- не сломал ли я то, что раньше работало

## Unit tests

Это тесты маленьких частей логики.

Пример:

- парсер команды
- функция подсчета статистики
- метод структуры

```rust
fn add(a: i32, b: i32) -> i32 {
    a + b
}

#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn adds_numbers() {
        assert_eq!(add(2, 3), 5);
    }
}
```

### Что здесь важно

- тест рядом с кодом
- маленький scope
- одна проверяемая идея

## Integration tests

Это тесты поведения системы целиком или нескольких модулей вместе.

Например:

- проверить CLI-сценарий целиком
- проверить, как несколько модулей работают вместе

На junior уровне полезно понимать отличие:

- unit test тестирует маленький элемент
- integration test тестирует взаимодействие

## CLI

CLI - это интерфейс через командную строку.

Пример:

```bash
todo add "buy milk"
todo list
todo done 2
```

CLI полезен как учебная форма, потому что заставляет:

- придумать команды
- описать аргументы
- валидировать ввод
- выдавать понятные сообщения об ошибках

## Работа с файлами

Как только программа должна что-то сохранять, появляется файловая система.

И здесь важно осознать:

- файл может не существовать
- формат может быть битым
- прав доступа может не быть

То есть файловые операции по своей природе уже требуют честной модели ошибок.

## Почему файловые операции естественно возвращают `Result`

Потому что неудача здесь не исключение в редком смысле, а нормальный возможный сценарий.

Если файл не открылся, приложение не должно:

- молча продолжать
- или паниковать без объяснения

Оно должно вернуть и обработать осмысленную ошибку.

## Сериализация

Сериализация - это перевод структуры данных в формат хранения или передачи.

На практике это может быть:

- JSON
- TOML
- YAML

Для учебного и junior-прикладного кода JSON часто самый удобный старт:

- читается человеком
- легко дебажить
- хорошо поддерживается

## `serde`

`serde` позволяет сериализовать и десериализовать структуры без ручного парсинга.

```rust
use serde::{Deserialize, Serialize};

#[derive(Serialize, Deserialize)]
struct Task {
    title: String,
    done: bool,
}
```

Это полезно потому, что:

- меньше рутины
- меньше мест для ручных ошибок
- типы и формат хранения связаны напрямую

## Почему ручной парсинг хуже, если задача типовая

Если ты вручную сериализуешь JSON, легко:

- забыть поле
- испортить формат
- не учесть крайний случай

Для типовых задач лучше пользоваться зрелым инструментом.

## Что делает CLI-проект хорошим

Минимально:

- команды понятны
- ошибки понятны
- состояние можно сохранить
- проект можно запустить по README
- есть тесты на важную логику

## Что нужно вынести из лекции

- тесты фиксируют поведение, а не просто украшают репозиторий
- CLI - хороший формат для учебного и портфельного проекта
- файловые операции естественным образом требуют `Result`
- `serde` снимает массу типовой работы и делает сериализацию надежнее
