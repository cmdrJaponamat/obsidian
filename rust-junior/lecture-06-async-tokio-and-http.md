# Лекция 6. Async, Tokio и HTTP: не магия, а способ переживать ожидание

## Почему async пугает

Новичок часто слышит:

- `async/await`
- `tokio`
- future
- runtime

и начинает воспринимать это как отдельную магическую подсистему языка. На самом деле тут нужно понять довольно земную вещь:

- что происходит, когда программа ждет

## Что такое blocking операция

Blocking операция - это операция, которая заставляет поток ждать.

Например:

- сетевой запрос
- чтение с диска
- ожидание таймера

Пока поток ждет, он не делает другую работу.

## Когда blocking становится проблемой

Если у тебя:

- один маленький CLI

то blocking может быть вообще нормальным выбором.

Если у тебя:

- много сетевых запросов
- много соединений
- много I/O-ожиданий

то держать отдельный поток на каждое ожидание может быть не лучшей моделью.

## Что такое async на интуитивном уровне

Async - это способ описать вычисление, которое может быть приостановлено в моменты ожидания и продолжено позже.

Не в смысле "само работает параллельно", а в смысле:

- пока одна задача ждет сеть
- runtime может выполнять другую готовую задачу

## Future

Когда ты пишешь `async fn`, ты по сути описываешь вычисление, которое возвращает future - объект, представляющий отложенный результат.

Важно:

- future сама по себе не равно "готовый результат"
- это скорее обещание вычисления, которое может быть завершено позже

## `await`

`await` означает:

- дождись завершения этой async-операции
- получи ее результат

Но это "дожидание" встроено в модель runtime, а не просто в блокировку потока привычным образом.

## Зачем нужен runtime

Runtime вроде `tokio` нужен потому, что:

- futures надо где-то исполнять
- нужно управлять тем, какие задачи готовы продолжаться
- нужны готовые I/O-примитивы для async-среды

То есть runtime - это не "библиотека ради библиотеки", а среда, которая вообще позволяет async-программе жить.

## Когда async действительно нужен

Async особенно хорош:

- в сетевых клиентах
- в сетевых серверах
- в I/O-нагруженных системах

## Когда async избыточен

Если задача:

- маленькая
- синхронная
- не упирается в I/O

то async может только усложнить код.

Это важный вывод: async не “более взрослый код по умолчанию”. Это инструмент под конкретные сценарии.

## HTTP как хороший пример

HTTP-клиент отлично подходит для учебной async-задачи, потому что объединяет:

- ожидание сети
- десериализацию ответа
- обработку ошибок
- таймауты

## Типичный сценарий

1. сделать запрос
2. получить ответ
3. проверить статус
4. распарсить JSON
5. сохранить или вывести результат

## Почему сетевые ошибки - это нормальный сценарий

Сеть может:

- не ответить
- ответить слишком долго
- вернуть ошибочный код
- вернуть неожиданный формат данных

Значит, сетевой код нельзя писать в модели "если повезет, будет `Ok`".

## Таймауты

Без таймаута сетевой код может висеть слишком долго.

С точки зрения пользователя это выглядит как:

- программа зависла
- непонятно, жив ли процесс

Поэтому таймауты и явная обработка ошибок - часть нормального сетевого дизайна.

## Что нужно вынести из лекции

- async нужен в первую очередь для задач с ожиданием I/O
- `async fn` возвращает future, а не "магически все делает сразу"
- runtime нужен, чтобы futures действительно исполнялись
- HTTP и async хорошо сочетаются как учебный пример
- таймауты и сетевые ошибки надо проектировать как нормальное поведение
