# Лекция 6. Async, Tokio, HTTP и где здесь настоящая польза

## Почему эта тема важна, но ею легко злоупотребить

Во многих Rust-вакансиях встречаются:

- `tokio`
- `async/await`
- HTTP
- JSON API

Из-за этого легко попасть в ловушку: начать использовать async везде подряд, не понимая, зачем он нужен.

Правильная цель этой лекции:

- понять, что такое async на интуитивном уровне
- понять, где он полезен
- понять, где он вообще не нужен

## Что такое blocking операция

Blocking операция - это операция, которая занимает поток ожиданием.

Например:

- ожидание ответа по сети
- ожидание чтения с диска

Пока поток ждет, он не делает другую работу.

## Почему это может быть проблемой

Если у тебя приложение должно:

- обслуживать много запросов
- ждать много сетевых ответов
- одновременно работать с большим числом I/O-задач

то блокирующая модель может использовать поток неэффективно.

## Что такое async на интуитивном уровне

Async позволяет описывать вычисление, которое может быть приостановлено в моменты ожидания и возобновлено позже.

Ключевая идея:

- когда задача ждет I/O, runtime может в это время исполнять другую задачу

Это не магия и не "ускорение всего подряд". Это другой способ организовать ожидание.

## `async fn`

Когда ты пишешь:

```rust
async fn fetch_data() { ... }
```

ты не получаешь "обычную функцию, которая сразу все делает". На практике такая функция возвращает future - описание вычисления, которое может быть выполнено позже.

## Что делает `await`

`await` говорит:

- дождись завершения этой async-операции
- получи ее результат

Но важно: все это работает в контексте runtime.

## Зачем нужен runtime

Сам по себе async-код не начинает жить автоматически.

Нужен runtime, например `tokio`, который:

- управляет выполнением async-задач
- дает инфраструктуру для сети, таймеров и других I/O-операций

Если говорить совсем грубо, runtime - это среда исполнения async-программы.

## Когда async действительно нужен

Async особенно полезен:

- в сетевых клиентах
- в сетевых серверах
- в задачах с большим числом ожиданий I/O

## Когда async может быть лишним

Если программа:

- маленькая
- однопользовательская
- в основном делает CPU-вычисления

то async может только усложнить код.

Это важный момент. Async - не знак "профессиональности". Это инструмент для конкретного класса задач.

## HTTP как хороший учебный пример

Очень полезный сценарий:

1. сделать запрос к API
2. получить JSON
3. десериализовать данные
4. отфильтровать их
5. сохранить результат

Этот сценарий хорош тем, что объединяет:

- сеть
- async
- `serde`
- обработку ошибок
- файловые операции

## Что важно понимать про HTTP

На интуитивном уровне:

- клиент делает запрос
- сервер отвечает
- ответ содержит статус, заголовки и тело

Тебе не нужно сразу становиться сетевым инженером, но нужно понимать:

- запрос может не дойти
- сервер может ответить ошибкой
- JSON может не совпасть с ожиданиями
- операция может зависнуть без таймаута

## Почему таймауты важны

Если сеть отвечает слишком долго, программа не должна ждать бесконечно.

Без таймаутов даже внешне простое приложение может:

- повиснуть
- вести себя как "сломанное"
- быть неудобным для пользователя

## Какие ошибки типичны в сетевом коде

- не удалось установить соединение
- истек таймаут
- сервер вернул ошибочный статус
- не удалось распарсить JSON

Именно поэтому сетевой код почти всегда должен быть `Result`-ориентированным.

## Что нужно вынести из лекции

- async нужен в первую очередь для I/O-нагруженных сценариев
- `tokio` дает runtime для async-кода
- HTTP-клиент - хороший практический пример применения async
- сетевые ошибки и таймауты надо проектировать как нормальную часть поведения, а не как исключение из мира
