# Лекция 7. Инженерные практики Rust-проекта: от “работает” к “поддерживается”

## Почему хороший код - это не только логика

Программа может быть функционально корректной, но неудобной для поддержки. На практике это проявляется так:

- код трудно читать
- ошибки обрабатываются хаотично
- проект неудобно запускать
- при любой правке страшно что-то сломать

Поэтому зрелость проекта определяется не только решенной задачей, но и инженерными практиками.

## `cargo fmt`

Форматирование нужно автоматизировать.

Почему:

- код становится единообразным
- диффы чище
- меньше ручных правок ради стиля

Это не косметика, а способ снизить трение при чтении и ревью.

## `cargo clippy`

Clippy помогает заметить:

- сомнительные идиомы
- лишние преобразования
- типичные недочеты

Это полезно особенно для junior, потому что ряд плохих шаблонов поведения еще не чувствуется интуитивно.

## Ошибки как часть дизайна

Ошибка - это не досадная помеха, а часть модели программы.

Нужно заранее понимать:

- какие ошибки вообще возможны
- где их надо обработать
- где их лучше передать выше

## `thiserror` и `anyhow`

### `thiserror`

Полезен, когда ты хочешь описать свой тип ошибки явно.

Это хорошо для:

- библиотек
- приложений с несколькими хорошо различимыми источниками ошибок

### `anyhow`

Удобен, когда тебе важен удобный верхнеуровневый pipeline ошибок.

Например:

- CLI-приложение
- быстрое прототипирование приложения

Нужно не заучивать “что моднее”, а понимать роль каждого инструмента.

## Логирование

Логирование нужно, чтобы видеть:

- что программа делала
- где произошла ошибка
- какие внешние условия были важны

Но логирование полезно только тогда, когда оно осмысленно.

Плохой стиль:

- беспорядочные `println!`

Лучший стиль:

- логировать важные события
- логировать причины сбоев
- не превращать код в шум

## README

README нужен не ради формальности, а чтобы другой человек мог:

- понять назначение проекта
- собрать его
- запустить
- протестировать

Без README даже хороший код становится хуже как проект.

## `clone()` как запах

В Rust избыточные `clone()` часто означают:

- ownership не до конца понят
- структура данных неудачна
- borrow checker “подавлен”, но не понят

Это не значит, что `clone()` всегда плох. Это значит, что он должен быть осознанным.

Полезный вопрос:

- зачем именно тут нужна копия

Если ответа нет, это повод пересмотреть дизайн.

## Что нужно вынести из лекции

- инженерное качество проекта определяется не только корректностью, но и удобством сопровождения
- `fmt` и `clippy` должны быть частью регулярного процесса
- ошибки, логи и README - это часть продукта, а не дополнение
- избыточные `clone()` часто указывают на непонятое владение
