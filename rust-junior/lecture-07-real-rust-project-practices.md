# Лекция 7. Практики реального Rust-проекта

## Зачем нужен этот слой

Можно написать код, который "как-то работает", и можно написать код, который выглядит как проект разработчика. Разница обычно в инженерных привычках.

## `cargo fmt`

Форматирование должно быть автоматическим.

Плюсы:

- меньше споров о стиле
- код проще читать
- diff чище

## `cargo clippy`

Clippy помогает находить:

- подозрительные конструкции
- лишние преобразования
- типичные недочеты

На junior уровне привычка прогонять `clippy` очень полезна.

## Ошибки: `thiserror` и `anyhow`

Важно понимать разницу ролей.

Упрощенно:

- `thiserror` удобно для определения своих типов ошибок
- `anyhow` удобно для приложений и верхнего уровня, где важнее удобная агрегация ошибок

Нужно не просто "чтобы компилировалось", а чтобы ошибка несла смысл.

## Логирование

Логи полезны для:

- диагностики
- понимания хода программы
- расследования нештатных сценариев

Но логирование должно быть содержательным.

Плохо:

- бессмысленные сообщения
- поток `println!` без структуры

Лучше:

- логировать важные события и причины ошибок

## Конфигурация

Даже маленький проект может иметь:

- переменные окружения
- путь к конфигу
- режимы запуска

Это быстро приближает учебный проект к рабочему.

## README

Нормальный README отвечает на вопросы:

- что делает проект
- как собрать
- как запустить
- как прогнать тесты
- какие ограничения есть

## Избыточные `clone()`

Это отдельная проблема Rust-новичков.

Если в коде много `clone()`, это повод спросить:

- правда ли копия нужна?
- нельзя ли передать ссылку?
- где на самом деле должен быть владелец?

## Имена и структура

Хороший Rust-код на junior уровне:

- не переусложнен
- функции короткие
- модули разделены по ответственности
- типы названы по смыслу

## Что обычно уже делает проект зрелее

- `fmt`
- `clippy`
- тесты
- типизированные ошибки
- README
- логирование

Это недорогие улучшения с большой отдачей.
