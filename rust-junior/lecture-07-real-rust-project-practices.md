# Лекция 7. Что делает Rust-проект взрослым: fmt, clippy, ошибки, логирование

## Почему "код работает" еще не значит "код хороший"

Junior очень часто оценивают не только по тому, решил ли он задачу, но и по тому, как выглядит путь от исходников до поддерживаемого проекта.

Проект становится заметно взрослее, когда в нем есть:

- форматирование
- линтинг
- нормальная работа с ошибками
- логирование
- понятный README

## `cargo fmt`

Форматирование должно быть автоматическим.

Почему:

- меньше ручной рутины
- меньше споров о стиле
- код легче читать
- diff чище

Это не "косметика". Форматирование напрямую влияет на удобство чтения и ревью.

## `cargo clippy`

Clippy - это линтер, который замечает:

- подозрительные конструкции
- лишние преобразования
- неудачные идиомы

Он не заменяет понимание языка, но хорошо помогает вычищать сырой код.

## Почему линтер важен для junior

Потому что он часто показывает:

- где код неидоматичен
- где можно написать проще
- где есть подозрительное место, которое ты сам еще не замечаешь

Привычка запускать `clippy` - один из дешевых способов быстро повысить качество кода.

## Работа с ошибками

Код без честной модели ошибок быстро становится хрупким.

Нужно понимать две вещи:

- ошибка - это нормальный сценарий
- ошибку нужно либо обработать, либо передать вверх осмысленно

## `anyhow` и `thiserror`

На практическом уровне полезно понимать роли:

### `thiserror`

Хорошо подходит, когда ты хочешь определить свой понятный тип ошибки.

### `anyhow`

Удобен на уровне приложения, когда тебе важнее удобно собирать и поднимать ошибки вверх.

На старте не нужно усложнять. Но нужно видеть, что ошибка - это часть архитектуры программы, а не досадная мелочь.

## Почему плохи бессмысленные `unwrap()`

`unwrap()` удобен как временная заглушка, но если он остается в важных местах, проект становится хрупким.

Хороший вопрос:

- что увидит пользователь, если операция не удалась?

Если ответ: "panic и стек-трейс", это часто плохой пользовательский опыт.

## Логирование

Логи помогают:

- понять ход выполнения
- диагностировать ошибку
- увидеть внешние условия программы

Но логирование должно быть содержательным.

Плохо:

- бесконечный поток `println!` без структуры

Лучше:

- фиксировать важные события
- логировать причины ошибок
- не шуметь без необходимости

## README

README отвечает на вопросы другого разработчика или твоего будущего "себя через месяц":

- что это за проект
- как его собрать
- как его запустить
- как его протестировать
- какие ограничения есть

Если README нет, проект становится заметно менее полезным.

## Избыточные `clone()`

В Rust это очень характерный запах плохого понимания ownership.

Конечно, `clone()` иногда нужен. Проблема возникает тогда, когда он:

- стоит почти везде
- используется как способ "заткнуть" borrow checker

Правильный вопрос:

- почему тут вообще требуется копия

Если ответа нет, это повод пересмотреть модель владения.

## Что нужно вынести из лекции

- качество проекта определяется не только логикой, но и инженерными практиками
- `fmt` и `clippy` - часть нормального рабочего процесса
- ошибки должны быть осмысленно встроены в проект
- логирование и README делают проект удобным для поддержки
