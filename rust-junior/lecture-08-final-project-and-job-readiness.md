# Лекция 8. Финальный проект и готовность к junior Rust

## Зачем нужен финальный проект

По Rust особенно важно иметь не только теорию, но и реальный код. Вакансий меньше, чем у C++, поэтому портфолио и демонстрация практики значат больше.

## Какой проект подходит

На уровне junior/intern достаточно проекта, который:

- решает понятную прикладную задачу
- имеет нормальную структуру crate
- использует `Result`
- имеет тесты
- имеет README
- при необходимости использует `serde` и async

Подходящие варианты:

- `todo-sync-cli`
- `expense-tracker-cli`
- `http-data-fetcher`
- `log-parser`

## Что показывает хороший Rust-проект

- ты понимаешь ownership, а не воюешь с ним
- умеешь строить CLI
- умеешь работать с файлами и ошибками
- умеешь использовать экосистему по делу
- можешь довести проект до воспроизводимого состояния

## Что важно не делать

- не пытаться строить "идеальную архитектуру"
- не добавлять лишние библиотеки ради количества
- не делать scope слишком большим

Для первого рыночного проекта важнее закончить и объяснить, чем бесконечно расширять.

## Как строить проект

Практичный порядок:

1. выбрать минимальный полезный сценарий
2. описать команды и данные
3. реализовать core logic
4. добавить работу с файлами или API
5. покрыть важные места тестами
6. вычистить ошибки, README и структуру

## Что нужно уметь рассказать на интервью

- что делает проект
- почему выбраны эти типы и crate
- где ownership, а где borrowing
- как обрабатываются ошибки
- как проект запускать и тестировать

## Признаки готовности подаваться

- ты понимаешь `String`, `&str`, ownership и borrowing
- можешь читать код с `Option` и `Result`
- умеешь пользоваться Cargo, fmt, test, clippy
- имеешь хотя бы один законченный проект
- можешь объяснить свой код спокойно и последовательно

Это уже достаточно, чтобы пробовать отклики на junior/intern роли.

## Что делать после Rust трека

Следующий слой усиления:

- глубже pet-project
- Docker
- SQL/PostgreSQL
- базовые backend-практики

Но сначала стоит реально закончить текущий проект и получить по нему ревью.

## Вопросы для самопроверки

1. Почему завершенный проект важнее большого, но незаконченного?
2. Что ты расскажешь про ownership в своем проекте?
3. Какие crate ты использовал и почему?
4. Готов ли твой проект к показу работодателю?
